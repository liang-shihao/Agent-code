00先建立一个心智模型

不是「更会聊」，而是「能动手」

编码 agent 的内核是「读文件 / 改代码 / 跑测试」的工具循环；办公 coworker 的内核几乎相同，只是把工具换成了整台电脑——它像人一样截屏看界面、移动光标、敲键盘、跨软件操作，或在有 API 时直接调用连接器。理解这个「感知—推理—行动」闭环，后面的一切（浏览器/桌面操控、连接器、技能、权限、记忆、多智能体、云端异步）都只是给它做加固。

从「用电脑」到「指挥电脑」。这是腾讯给 WorkBuddy 的官方叙事，也精准概括了整个品类的范式跃迁：Anthropic 给 Claude Cowork 的价值主张是「描述一个结果，走开，回来时工作已经完成」。二者共同点在于——用户不再逐步操作软件，而是把一个结果目标交给 agent，由它自主拆解、多步执行、必要时暂停向人请示，最后交付可直接使用的成品（报表、表格、PPT、审批、简报）。这就是「数字员工 / 数字同事」这个词的实质。

# 通用办公 agent 的内核——computer-use 感知-行动循环
state = [system_prompt, user_goal]
while True:
    frame  = capture_screen()                       # 感知：截图（可叠加无障碍树/DOM）
    resp   = model.create(state + [frame],          # 多模态模型看屏幕，输出下一步
                          tools=[computer, bash, editor, mcp_tools])
    state.append(resp)
    actions = [b for b in resp if b.type=="tool_use"]
    if not actions: break                          # ★ 无动作 = 任务完成 / 交回用户
    if is_risky(actions):                           # ★ HITL 门：发送/付款/删除等需人批准
        await human_approve(actions)
    results = [execute(a) for a in actions]         # 点击 (x,y) / 输入 / 调 API
    state.append(observe(results))                  # 观察结果（新截图）回流
    if steps >= max_steps or over_budget: break    # ★ 兜底：步数 / 预算 / 时钟

三个贯穿全篇的关键事实：① 与编码 agent 一样，终止大多是隐式的——模型给出一轮「不再请求动作」的回复即代表完成；② 这一代产品的分水岭不是「聊天质量」，而是能否可靠地把屏幕像素/DOM 映射成正确坐标动作、跑完几十上百步而不漂移、并在危险处停下等人；③ 真正把 agent 从玩具变成「数字员工」的门槛，是接入企业系统（连接器/MCP）+ 权限治理 + 长时后台运行——纯 demo 谁都能跑，能进营销/财务/人事/客服的真实流程才算数。

坐标 grounding 是最硬的底层难题。Anthropic 明确指出「训练 Claude 精确地数像素至关重要，否则模型给不出可用的鼠标指令」。纯视觉路线（看渲染像素、预测点击坐标）通用性强但一次浏览器任务可能烧掉约 5 万 vision token；而解析 DOM/无障碍树只需约 5000 text token 且更稳——成熟系统因此走「DOM/无障碍树优先、视觉兜底」的混合架构。
012022 → 2026 · 四代范式叠加

从「会说」到「会做」，再到「替你上班」

Chatbot：会说话的大脑

2022 末 · 起点

ChatGPT 点燃全民认知，但它只是一个无状态、无工具、碰不到你文件的对话模型。与此同时，Adept 的 ACT-1 已在演示「用 Transformer 操作浏览器与软件」——为两年后的 computer-use 埋下种子。

2022-09
Adept 发布 ACT-1（Action Transformer） ——首次公开演示模型可按自然语言操作浏览器/软件，通用数字 agent 的最早雏形。
2022-11-30
OpenAI 发布 ChatGPT ——5 天破百万用户，定义了「Chatbot」这一代，也让后续所有 agent 有了可用的底座模型。

Copilot：坐进软件的副驾驶

2023

微软把 GPT-4 嵌进 Office 全家桶，「Copilot」成为这一代的代名词——它在你身边建议、起草、补全，但人始终握着方向盘做每一步。同期 AutoGPT/BabyAGI 掀起「放它自己跑」的全自主热潮，验证了自主循环可行，却也暴露了脆弱与烧 token。

2023-03-16
Microsoft 365 Copilot 公布 ——GPT-4 嵌入 Word/Excel/PPT/Teams/Outlook，2023-11-01 面向企业 GA（$30/人/月），确立 Copilot 范式。
2023-03
BabyAGI / AutoGPT 热潮 ——AutoGPT（3-30 发布）数周破 10 万 GitHub 星，普及「目标→规划→执行→重排」的自主循环想象，但落地脆弱。

Agentic coworker：永不下班的数字同事

2025H2 → 2026

两条线开始合流，TechCrunch 称之为「编码 agent 的战争正溢出到办公室的其余部分」，竞争焦点从「谁聊得好」转向「谁拥有工作真正发生的场所」。前沿实验室把编码 agent 的血统复用给白领市场，中国厂商则以「本地执行 + 办公全家桶」切入并反向出海。Anthropic 对 120 万次 Cowork 会话的分析显示：90%+ 用途并非编码（业务流程操作 33.4%、内容创作 16.4%、软件开发仅 8.7%）——实证了 coworker 线已成主战场。

产品全景：谁在做，能力如何，凭什么赢

同一个「感知-行动」内核，被三股力量包装成不同形态：焦点双雄（WorkBuddy / Cowork）、海外大厂与独立玩家、国内大厂与模型公司。切换标签查看每一阵营；每张卡片给出定位、代表能力与差异化。2026 年两个行业信号值得注意：① 独立「agent 浏览器」退潮——OpenAI 弃 Atlas、Google 关 Mariner，浏览能力被折叠回主力助手；② 企业主战场转向「平台 + 治理 + 舰队管理」而非单点 agent。

03拆开外壳 · 9 大组件

架构原理：数字员工是怎么「动手」的

数字员工的本质是一个感知—推理—行动闭环，外挂工具、连接器、记忆与编排层。这九层大体互相独立，可逐个理解、逐个加固。每个组件都给出「是什么 / 怎么工作 / 关键抉择 / 常见陷阱 / MVP vs 成熟版」。点击展开。

01 Computer-use / OS 操控 截图 → VLM 理解 → 鼠标键盘动作 → 再截图验证 → 循环 +

是什么

让模型像人一样操作电脑的核心循环。Anthropic 于 2024-10 随 Claude 3.5 Sonnet 首发 computer use，工具族为 computer + bash + text_editor；截至 2026 年现行 beta header 为 computer-use-2025-11-24（支持 Claude Sonnet 5、Opus 4.8/4.7/4.6/4.5、Sonnet 4.6）。

怎么工作 · 坐标 grounding 是命门

模型分析截图后必须计算出精确像素坐标才能下达 click(x,y)。Anthropic 明确「训练模型精确数像素至关重要」；OpenAI CUA 则把坐标计算放在其托管虚拟浏览器服务端。两条纯视觉路线的公开对比很能说明难度：OSWorld 上 OpenAI CUA 38.1% vs Anthropic Computer Use 22%（人类基线 72.4%）；WebVoyager 上 CUA 87% vs Anthropic 56%。

关键设计抉择

屏幕感知方式：纯视觉像素（通用、能对付 Canvas/WebGL，但 vision token 贵 3–5 倍）vs 解析 DOM/无障碍树（便宜可靠但对自绘组件失效）。共识是混合——DOM 优先、视觉兜底。
动作粒度：低层坐标点击（通用但易漂移）vs 语义元素点击（给可交互元素编号，模型输出编号而非坐标，更稳）。

常见陷阱

  • 坐标漂移：模型按逻辑分辨率给坐标、执行按物理像素，高分屏必须做 DPI 缩放换算。
  • 每步塞入全部历史截图会瞬间撑爆上下文——必须做截图窗口裁剪（只留最近 N 张）。
  • 不设步数上限：官方 demo 的循环本身不内置迭代上限，靠调用方兜底，漏加就会无限烧钱。
MVP

单一 VLM 循环、固定分辨率截图、纯坐标点击、默认 ~10 轮上限、无自我纠错。直接抄 Anthropic computer-use-demo 的 loop.py 即可跑通。

成熟版

多模态融合截图 + 无障碍树、坐标预测经专门训练、动作前置安全暂停、动态分辨率与元素高亮（set-of-marks）、失败自我纠错重试、截图窗口裁剪保 prompt cache。

02 浏览器操控 vs 桌面操控两条路径 视觉像素 · DOM 解析 · 无障碍树——各有取舍 +

是什么

「办公 = 在 SaaS 里点点点」的部分走浏览器路径；操作 Excel、ERP、旧式客户端等无 API 的本地应用走桌面路径。能操作任意本地应用正是「数字员工」区别于纯浏览器 agent 的核心。

怎么工作 · 三种浏览器感知

①视觉/像素（Anthropic Computer Use、OpenAI CUA）：看渲染像素、预测点击坐标，通用但贵，一次任务可能烧 5 万 vision token。②DOM/无障碍树（Playwright MCP、Stagehand、browser-use）：把上千 DOM 节点坍缩成有语义的可交互集合（按钮/链接/表单），更便宜可靠。③桌面 a11y 树（Windows UIA / macOS AX / Linux AT-SPI）：操作本地应用。

关键设计抉择

路径选择：能用 API 就别用 GUI，能用 DOM 就别用视觉——把纯视觉留给 Canvas/自绘组件与无 API 的长尾系统。
回放 vs 推理：高频固定流程用「录一次、回放」（如 OpenAdapt，健康回放时零模型调用），比每次从头推理省钱，屏幕变化时再用模板匹配/OCR 兜底。

常见陷阱

  • 纯 DOM 选择器一改版就崩；iframe/shadow DOM/动态加载元素难捕获。
  • 反自动化检测（登录态、验证码）阻断浏览器路径。
  • 桌面纯坐标回放极脆——必须锚定 a11y 元素或图像模板，否则窗口焦点/分辨率一变就失败。
MVP

单一路径（多为 DOM），脆弱选择器，覆盖固定几个网页流程。

成熟版

混合架构——优先 DOM/无障碍树，缺失时回退视觉；OpenAI CUA 本身也叠加截图 + DOM + 无障碍树解析；桌面与浏览器双通道，高频流程录制自愈。

03 连接器与 MCP（Model Context Protocol） 让 agent 安全接入企业系统、工具与数据的开放协议层 +

是什么

MCP 是 Anthropic 2024-11 开源的开放标准，用统一协议消除 N×M 集成复杂度，让 agent 安全地发现、调用企业工具/API/数据（JSON-RPC 2.0、有状态连接）。2025-12-09 Anthropic 将 MCP 捐给 Linux 基金会旗下新成立的 Agentic AI Foundation（AWS/Google/微软/OpenAI 背书），坐实其成为「agent 工具的 USB-C」。

怎么工作 · 四段循环

agent 作为 MCP Host/Client 连一批 MCP Server：schema 注入 context → 模型选工具 → 确定性代码执行 → 结果回注 context 形成反馈闭环。2025-06 规范把 MCP server 定义为 OAuth 2.1 资源服务器，要求客户端实现 Resource Indicators（RFC 8707）；Streamable HTTP 成为远程生产级传输标准（多并发、水平扩展、企业 OAuth）。分工上：MCP 负责「协议层/工具发现调用」，Connector/API 是具体后端实现（Atlassian、Salesforce、SAP 已出产品级 MCP 连接器）。

关键设计抉择

GUI 操控 vs 直连 API：有 API 优先直连（快、稳、可鉴权），GUI 操控留给没有 API 的系统。WorkBuddy 企业版正是基于腾讯云 One ID 通过 Connector 接 OA/CRM/项目管理，并支持 MCP。
工具数量：工具太多会导致选择困难、context 膨胀——需按需加载/分组。

常见陷阱

  • MCP Server 是新攻击面（prompt injection、越权），需配凭证隔离与审计（见组件 05）。
  • 本地 stdio server 无鉴权，直接上生产 = 越权风险。
MVP

本地 stdio server、无鉴权、单用户，接几个只读工具。

成熟版

远程 Streamable HTTP + OAuth 2.1 + MCP Gateway + 审计 + 凭证隔离；企业就绪（SSO、审计轨迹、配置可移植）。

04 技能生态（Agent Skills / OpenClaw Skills） 把可复用工作流打包成可移植文件夹，按需自动加载 +

是什么

Skills 把可复用工作流、领域知识、新能力打包成可移植文件夹（含 SKILL.md 指令 + 脚本 + 资源），agent 按需自动发现调用。Anthropic 于 2025-10-16 首发 Agent Skills，并于 2025-12-18 正式作为开放标准发布（规范 + SDK），号称同一 Skill 可在 Claude.ai / Claude Code / API，以及 Codex、Gemini CLI、Copilot、Cursor 等 20+ 平台通用。OpenClaw 是与之并行的开放 Agent Skills 生态；腾讯 WorkBuddy 兼容 OpenClaw 规范。

怎么工作 · 渐进式披露

启动时只读每个 skill 的 name + description（约 100 token/个），任务匹配才加载正文（workflow、脚本、schema、API 文档）——用极小的常驻开销换按需的完整能力。腾讯为中国用户建的 SkillHub 是 OpenClaw 官方 ClawHub 的本土化高速镜像，已聚合超 1.3 万个 Skills，并将 10+ 腾讯自有产品做了 skill 化。注：网传「腾讯自有 7 万 Skills」为讹传，已按镜像口径更正

关键设计抉择

常驻 vs 按需：把所有指令塞进 system prompt（简单但吃 token、稀释指令遵从）vs 渐进式披露（只在需要时加载）。成熟系统一律选后者。

常见陷阱

  • description 写不好 → 该触发时不触发。
  • 技能里嵌可执行脚本 = 供应链风险，需审计与沙箱。
MVP

单个 prompt 片段、手动挂载。

成熟版

渐进式披露、公司级 Skill 库、跨平台标准化分发、使用即沉淀（流程自动转化为可复用 Skill，WorkBuddy 的「知识飞轮」）。

05 权限、沙箱、审计与人类在环（HITL） 让 agent「安全地」动真实系统——危险处停下等人 +

是什么

把「能跑 demo」变成「敢进生产」的那一层。包含四件事：沙箱隔离、凭证隔离、权限模型、HITL 审批门 + 不可篡改审计。这是数字员工进入财务/人事等高风险流程的准入条件。

怎么工作

沙箱：Manus 把 agent 放进完全隔离的 Ubuntu 虚拟机；2026 主流用 MicroVM / gVisor 做一次性可弃环境。凭证隔离：Anthropic Managed Agents 让凭证绝不进入沙箱——OAuth token 存 secure vault，仅经 harness 看不见的 MCP proxy 访问。HITL：金融交易、数据删除、对外发送等不可逆动作需显式人类批准，审批清单含意图、数据血缘、权限链、影响半径、回滚方案。Cowork 的 Manual/Auto/Skip 三档、WorkBuddy 的「任务拆解后需用户确认 + 自动拦截高危操作」都是这层的产品化。

关键设计抉择

权限 vs 沙箱是两道不重叠的闸：权限决定执行前「能不能做」，沙箱决定执行后「能碰什么」，两者都要。
强制下沉：prompt 级 → 应用级 → OS/内核级。安全关键规则往下迁，别信任模型配合。

常见陷阱

  • 确认疲劳（confirmation fatigue）——批得太频繁，用户就闭眼点同意，HITL 形同虚设。
  • HITL Dialog Forging 攻击：伪造确认框内容骗批准——确认内容必须可信渲染、不可被 agent 篡改。
  • 间接提示注入：把恶意指令藏进文件注释/工单描述/API 返回等数据通道，agent 当合法指令执行真实动作。
MVP

本地直接执行、无沙箱、危险动作一律 y/n 确认、基本日志。

成熟版

一次性可弃沙箱 + 凭证保险库 + 不可篡改审计 + 分级 HITL + 反向模型 sanity check；符合 EU AI Act 第 14 条的可干预/停止/覆盖接口。

06 数字员工模型（身份 / 权限 / 记忆 / 长任务） 脑—手解耦：让 agent 有持久身份、崩溃可续、独立伸缩 +

是什么

把一次性会话升级为有身份、有记忆、能长时运行的「员工」。Anthropic Managed Agents 的脑—手解耦是代表架构：把「脑」（模型 + harness）与「手」（沙箱 + 工具）分离，各自独立伸缩与故障恢复。

怎么工作 · 三大虚拟化组件

Session：append-only 事件日志，durable 存于 harness 之外（emitEvent/wake/getSession 恢复）。Harness：无状态编排循环，崩溃后从最后事件续跑。Sandbox：可弃执行环境。效果：容器按需供给、推理无需等容器初始化，p50 TTFT 降约 60%、p95 降超 90%。「Many brains, many hands」——多 harness 连多 sandbox，脑可互传手。

常见陷阱

  • 无持久身份 → 任务失败即丢、无法审计追责。
  • 长任务的登录态/会话保活未处理，跑一半掉线。
MVP

无持久身份、任务失败即丢、单进程。

成熟版

持久 Session 作上下文对象、任意切片回放、跨设备恢复、企业身份/权限绑定、崩溃续跑。

07 多 Agent 编排与并行 编排者-工人模式：拆任务、并行执行、综合校验 +

是什么

用「规划 + 执行 + 验证」分工处理复杂长任务。主流是 orchestrator-worker（编排者-工人）：lead agent 拆任务，并行 spawn N 个各带独立上下文、窄系统提示的 subagent，收集输出后综合。WorkBuddy 的「多 Agents 并行」、Manus 的多 agent 架构、Anthropic 多 agent 研究系统都是此模式；2026 层面还叠加 A2A（Agent-to-Agent）协议。

怎么工作

planner 拆解 → 多 executor 并行/串行执行 → verifier 校验结果，失败回退重规划。并行 subagent 各带独立上下文可降低幻觉、提高覆盖。企业对多 agent 的关注度暴涨——Gartner 称相关咨询量从 2024 Q1 到 2025 Q2 增长 1445%（客户问询量，非部署量）。

常见陷阱

  • agent 间上下文传递丢失、错误级联。
  • 成本随 agent 数增长——数据未证明复杂性之前，优先单 agent。
MVP

单 agent 串行。

成熟版

并行隔离上下文、消息传递协作、确定性工作流编排、A2A 跨 agent 互操作。

08 云端异步执行 vs 本地执行 合上笔记本任务继续跑——数字员工的核心体验 +

是什么

决定 agent 在「哪台机器、你在不在场」运行。这是 2026 年最能拉开体验的一维。

怎么工作

本地/桌面：用你的权限操作你眼前的机器（读写文件、跑 shell）。Anthropic Dispatch（2026-03-17 研究预览）把手机任务路由到本机 Claude Desktop 执行。WorkBuddy 桌面端直接访问本地文件。云端/远程：Claude Cowork 于 2026-07-07 走出桌面 App，会话在 Anthropic 服务器远程执行并存到账户——工位起任务、手机看进度、稍后取结果，关机后仍继续、定时任务无需在线设备。Manus/WorkBuddy 企业版的数字员工同样云端常驻。

常见陷阱

  • 长任务的登录态保活、成本与超时控制。
  • 失败任务的可恢复性（需配合组件 06 的持久 Session）。
MVP

仅本地同步、关机即停。

成熟版

云端持久会话 + 定时/调度 + 跨设备接力 + 本地/云混合，每任务独立云沙箱。

09 上下文与记忆 长任务不触顶、跨会话不失忆 +

是什么

长任务下工具结果/推理/消息累积会触顶 token 上限，且存在 context rot（token 越多、召回越差）。管理分两层：会话内的压缩（compaction）与跨会话的长期记忆。

怎么工作

压缩：接近窗口上限时把早期历史摘要成紧凑状态，agent 从摘要 + 近期上下文续跑。Anthropic 已在 Claude Opus 4.6 产品化平台侧自动上下文压缩（beta），目标做到「上下文感觉无限」。记忆：三层——工作记忆（当前会话）、情景记忆（完整事件流）、长期记忆（片段 → embedding → 向量库 → top-k 检索）；2026 前沿是图谱记忆（向量解决「相似性」、图谱解决「关系性」，支持多跳推理）。

常见陷阱

  • 只用 input_tokens 计量窗口——cache-creation/cache-read/待输出都占窗口，少算就硬溢出。
  • 记忆污染/过期事实、无遗忘导致检索噪声、PII 落库的隐私合规。
MVP

单一上下文窗口、满了就截断。

成熟版

自动 compaction + 外置 Session 事件日志 + 向量/图谱记忆 + 工具结果清理 + 缓存高效上下文管理。

04从零到成熟 · 10 个阶段

从零构建一个 WorkBuddy 类 agent

核心工程直觉：先把最小「感知-决策-执行」闭环跑通，再逐层加壳（浏览器 → 桌面 → 连接器 → 技能 → 沙箱/HITL → 记忆 → 多 agent → 云端 → 评估）。每个阶段都应可独立运行、可回归测试。参考实现来自 Anthropic computer-use-demo、browser-use、OpenAdapt、Manus。点击展开每阶段的目标 / 构建步骤 / 可复用选型 / 验收标准 / 常见坑。

0 最小 Perceive-Act 循环 截图 → 模型 → 单步动作 → 再截图，所有 computer-use agent 的原子内核 +
目标 · 跑通「截图 → 多模态模型 → 一个动作（click/type）→ 再截图」，内化控制结构，此刻不必有用。

构建步骤

  • 抓屏（mss/pyautogui）→ base64 编码 → 调用带 computer 工具的模型。
  • 模型返回 action + coordinate → 本地执行鼠标键盘 → 把新截图作为 tool_result 回传。
  • 循环直到模型不再请求工具即视为完成；加硬 max_steps 防死循环。
可复用选型 · 直接抄 anthropics/anthropic-quickstarts 的 computer_use_demo/loop.py——它演示了工具版本分组、异步工具执行、以及 _maybe_filter_to_n_most_recent_images()（只留最近 N 张截图省 token/保 prompt cache）。self-operating-computer 是更精简的纯视觉起点。

验收标准

  • 能在记事本里输入指定文字、点开一个桌面图标，循环干净退出、永不挂起。

常见坑

  • loop.py 官方实现不内置迭代上限，必须自加 max_steps。
  • 高分屏坐标缩放（逻辑分辨率 vs 物理像素）。
  • 每步塞全部历史截图会爆 context——务必裁剪图像窗口。
1 浏览器操控（DOM 优先） 覆盖「办公 = 在 SaaS 里点点点」的大部分场景 +
目标 · 让 agent 可靠操作网页（比盲点像素更稳）。

构建步骤

  • 每步把浏览器状态（URL、索引化的可交互元素列表/AXTree、DOM、截图）喂给模型。
  • 模型返回结构化输出：对上一步的评估 + memory 进度 + 下一子目标 + 动作（输出元素编号而非 CSS selector）。
可复用选型 · browser-use（Python/MIT，GitHub 约 10.6 万 star，WebVoyager 报告 89.1%；底层已从 Playwright 迁到直连 CDP + 事件驱动 watchdog 架构）。办公场景先 DOM 后视觉兜底最划算。

验收标准

  • 完成「登录内部系统 → 搜索 → 导出表格」这类 3-5 步网页流程。

常见坑

  • 纯 DOM 遇 canvas/自绘组件失效，需视觉 fallback。
  • 动态加载/iframe/shadow DOM 元素捕获；反自动化检测（验证码）。
2 桌面操控 + 录制式自愈 覆盖非浏览器的本地应用（Excel、ERP、旧系统） +
目标 · 操作无 API 的本地桌面软件——这是「数字员工」区别于纯浏览器 agent 的核心。

构建步骤

  • 接入 OS 级鼠标键盘 + 可访问性树（Windows UIA / macOS AX / Linux AT-SPI）。
  • 对高频重复流程引入「录一次、回放」，屏幕变化时用模板匹配/OCR 兜底。
可复用选型 · OpenAdapt——把一次人工演示编译成自愈脚本，健康回放时零模型调用（比每次从头推理省钱），配套 openadapt-agent 提供安全门/HITL/审计。这是「贵的通用 agent」与「便宜的确定性 RPA」之间的关键混合选型。

验收标准

  • 在无 API 的桌面软件里完成一段可回放的多步操作。

常见坑

  • 坐标漂移、窗口焦点丢失、跨分辨率/DPI。
  • 纯坐标回放极脆，必须锚定 a11y 元素或图像模板。
3 工具 / 连接器（MCP） 有 API 就直连 API，接企业系统（CRM、邮件、日历、DB） +
目标 · 把「靠 GUI 点」升级为「有 API 就直连」——快、稳、可鉴权。

构建步骤

  • agent 作 MCP Host/Client，连一批 MCP Server；四段循环：schema 注入 → 模型选工具 → 确定性执行 → 结果回注。
  • 建工具注册表与 schema 注入器；能用 API 就别用 GUI，GUI 留给没有 API 的长尾系统。
可复用选型 · MCP（Anthropic 2024-11 开放标准，2026 已是「任何认真用工具的 LLM 系统的基础设施」，已捐给 Linux 基金会 AAIF）。Atlassian/Salesforce/SAP 等已出产品级 MCP 连接器。

验收标准

  • 一句话「把这封邮件的要点建成日历事件」跨两个连接器完成。

常见坑

  • 工具过多导致选择困难/context 膨胀。
  • MCP Server 是新攻击面（prompt injection、越权），需在 Stage 5 加固。
4 技能系统（Skills） 把领域 SOP 沉淀为可复用、按需加载的知识 +
目标 · 避免把所有指令塞进系统提示，把流程沉淀成可插拔技能。

构建步骤

  • 采用 SKILL.md + 渐进式披露：启动只读 name+description（约 100 token/个），任务匹配才加载正文。
  • 建技能目录、YAML frontmatter 索引、按需加载器、可执行脚本挂载。
可复用选型 · Claude Agent Skills（2025-10 首发功能、2025-12-18 作为开放标准发布，文件系统式架构）；或兼容 OpenClaw 生态复用社区技能（WorkBuddy 路线）。

验收标准

  • 新增一个「报销审批 SOP」技能后，agent 能零改代码执行该流程。

常见坑

  • description 写不好 → 该触发时不触发。
  • 技能里嵌可执行脚本 = 供应链风险。
5 权限沙箱 + HITL（人在环） 危险动作前暂停等人批准，隔离执行、全程审计 +
目标 · 让 agent「安全地」动真实系统——这是进生产的准入条件。

构建步骤

  • 动作分级（只读/写/不可逆）；不可逆动作（发送、付款、删除）强制 HITL 确认。
  • 隔离执行环境（容器/独立 VM）+ 全程不可篡改审计日志；关键动作用「双因子判断」（人审 + 反向模型 sanity check）。
参考规范 · EU AI Act 第 14 条要求高风险系统提供可「干预、停止、覆盖」的人机接口；凭证隔离参考 Anthropic Managed Agents（凭证绝不进沙箱，经 MCP proxy 访问）。

验收标准

  • agent 试图发外部邮件时必弹确认并可拒绝；沙箱内操作不影响宿主。

常见坑

  • 确认疲劳——批得太频繁就闭眼点同意。
  • Dialog Forging：确认框内容被 agent 篡改骗批准，须可信渲染。
6 记忆 跨任务/会话复用偏好与事实，从「金鱼」变「老搭档」 +
目标 · 第二次接同类任务时无需重复告知偏好。

构建步骤

  • 三层：工作记忆（会话 sliding window）、情景记忆（完整事件流）、长期记忆（片段 → embedding → 向量库 → top-k 检索）。
  • 加巩固/索引/更新/遗忘的生命周期管理；进阶引入图谱记忆做多跳推理。
2026 前沿 · 图谱记忆崛起（向量管「相似性」、图谱管「关系性」）；MUSE 等框架演化出「规划-执行-反思-记忆」闭环。

验收标准

  • 第二次接同类任务时自动沿用已知偏好/事实。

常见坑

  • 记忆污染/过期事实、无遗忘导致检索噪声。
  • PII 落库的隐私合规。
7 多 Agent 编排 规划 + 执行 + 验证分工，处理复杂长任务 +
目标 · 一个需 10+ 步、跨 3 个系统的任务端到端交付。

构建步骤

  • planner 拆解 → 多 executor 并行/串行 → verifier 校验，失败回退重规划。
  • 编排框架可用 LangGraph 类状态机；跨 agent 用 A2A 协议互操作。
可复用选型 · Manus 是范式样板（规划/执行/验证 agent 协作，全程在独立 VM 内跑）。仅当实测出现漂移/协调失败时才加多 agent 仪式。

验收标准

  • 跨 3 个系统的 10+ 步任务端到端交付成品。

常见坑

  • agent 间上下文传递丢失、错误级联。
  • 成本随 agent 数指数增长。
8 云端异步执行 任务扔上去人可下线，跑完再推送——数字员工的核心体验 +
目标 · 关浏览器后台仍跑完并推送通知。

构建步骤

  • 每个任务分配独立云沙箱/VM，工具调用全在云端完成（无需本地装软件）。
  • 异步任务队列 + 完成推送 + 定时/调度；配合 Stage 6 的持久 Session 做可恢复。
可复用选型 · Manus 模式（任务全在云沙箱、离线异步、完成推送）；沙箱基座可用 E2B/Firecracker 类微 VM。Cowork 的云端后台运行是产品级参照。

验收标准

  • 关闭客户端后台仍跑完，结果推送到多设备。

常见坑

  • 长任务的会话/登录态保活、成本与超时控制。
  • 失败任务的可恢复性。
9 评估与回归 量化「改了 prompt/模型后是变好还是变差」，防回归 +
目标 · 每次发版产出可对比的「成功率 / 步数 / 成本」三联指标。

构建步骤

  • 搭 CI 化评估集：标准 benchmark + 自建业务任务集，多次运行取均值（单次报分会虚高 5-15 分）。
  • 把安全 red-team 与「带策略完成率」纳入回归，要求单调改善或给明确权衡理由。
可复用基准 · OSWorld / OSWorld 2.0（真实桌面、长程）、WebArena / WebVoyager（浏览器）、TheAgentCompany（真实办公）、GAIA、ST-WebAgentBench（安全）。详见下一节。

验收标准

  • 每次发版可对比的成功率/步数/成本三联指标，回归可复现。

常见坑

  • 数据污染与 scaffolding 抬分、单 run 报喜。
  • 只测成功率不测成本/步数与安全。
05现实检验 · 基准 · 风险

评估、基准与落地成效：远没到「换掉白领」

衡量数字员工的基准正从「单步准确率」转向「长程、真实环境、带安全约束的任务完成率」。一句话结论：Web 短任务已相当可用（~94%），但长程真实办公仍卡在 20–30% 的天花板——demo 惊艳与生产可靠之间隔着一条鸿沟。

企业落地该看的四个指标

01

任务完成率 vs 部分完成率。区分「全对」与「部分对」（TAC、OSWorld 2.0 都双报）——长程任务的「部分分」高、「完全完成」低，别被前者误导。

02

人工干预率 / 接管率。HITL 在关键决策必设。厂商宣称的「90% 零人工」多为自家口径，选型时要用自己的流程实测接管率。

03

单任务成本与时延。OSWorld 2.0 显示长任务动辄数百次工具调用，token/延迟是可用性瓶颈；更省 token 的模型往往完成率更低，须权衡。

04

带策略的安全完成率（CuP）。只有「完成且不违反任何策略」才算分——SOTA agent 启用策略后成功数大幅缩水，这是任务型基准完全测不到的缺口。

失败模式与安全风险

RISK 1 · 头号威胁

间接提示注入 / 目标劫持

OWASP 于 2025-12-09 发布《Top 10 for Agentic Applications 2026》，把 Agent Goal Hijacking（ASI01） 列为头号风险。把恶意指令藏进文件注释、工单描述、API 返回等数据通道，agent 会当合法指令执行真实动作——比直接注入更危险，因为它披着可信数据的外衣。

RISK 2

工具使用错误与幻觉执行

常见 7 类工具错误：调用不足、参数值/名/类型错误、重复调用、幻觉函数名、非法输出格式。表现为「结构合理但语义错误」的动作——比纯文本幻觉更危险，因为它会真的点下去、发出去。

RISK 3

规划漂移与失控成本

长任务中计划与执行背离、死循环烧 token、误操作不可逆资源。生产环境必配步数/预算/时钟上限 + 连续失败熔断，并把危险动作强制下沉到 HITL 与 OS 沙箱。

方法：冻结任务集 + red-team 套件，每阶段跑一遍，要求 solve-rate 与安全性单调改善（或给明确权衡理由），同时盯住成本/延迟；记录每个 action/observation 让运行可重放。对企业选型而言：盯 CuP 这类「带策略的完成率」、真实接管率、单任务成本，并把间接提示注入列为首要威胁——而非只看厂商发布会里的 demo 与自述 DAU。
