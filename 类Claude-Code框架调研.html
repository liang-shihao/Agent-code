一切都围绕一个循环

所有这类框架——无论装在终端、编辑器还是云端——本质都是同一个 ReAct 式循环：把不断增长的对话发给模型，执行它请求的工具，把结果追加回去，再重复，直到模型不再调用工具、直接给出文字答复为止。理解这个循环，后面的一切（工具、上下文、权限、多智能体）都只是给它做加固。

模型负责思考，harness 负责驾驶。模型是无状态的——它不记得上一轮，也碰不到你的文件系统。是外层的循环代码给了它记忆（把历史全量重发）、给了它双手（工具）、给了它安全边界（权限与沙箱），并决定何时停下。同一个聊天模型，仅仅因为一个循环不断带着新观察结果去重新调用它，就"变成"了智能体。

# 循环骨架——production 工具的内核也就是这几行
messages = [system, user_prompt]
while True:
    resp = model.create(messages, tools=tool_schemas)   # 全量历史每轮重发
    messages.append(resp.assistant_message)             # text + tool_use 块
    calls = [b for b in resp.content if b.type=="tool_use"]
    if not calls: break                              # ★ 主终止：没有工具调用 = 完成
    results = [run_tool(c) for c in calls]           # 每个 tool_use 必须有对应 tool_result
    messages.append(user_message(results))              # 观察结果作为 user 角色回流
    if turns >= max_turns: break                    # ★ 副终止：轮数 / 预算 / 时钟兜底

三个反直觉但关键的事实，会贯穿整份指南：① 终止是隐式的——模型给出一轮"没有工具调用"的回复即代表结束，默认没有显式的"完成"信号；② 工具结果以 user 角色回流（"环境以用户身份说话"）；③ 每一个 tool_use 都必须在下一条消息里有 id 严格匹配的 tool_result，漏一个或错位就是硬报错。这三点看似琐碎，却是 80% 新手 loop 崩溃的根因。

从"放它自己跑"到"实验室的第一方产品"

这类工具的历史，是一条从狂热的全自主实验，收敛到有纪律的工程实践，再被前沿实验室收编为第一方产品、并最终平台化的曲线。下面按 5 个时代梳理，每个时代下是关键里程碑。

自主智能体淘金热

2023 · 3–12 月

GPT-4 点燃了一切。AutoGPT、BabyAGI、GPT-Engineer 这些业余项目证明了"给 LLM 一个目标让它自己循环"是可行的，狂揽关注度，也暴露了死循环与烧 token 的通病。与此同时，Aider 与 Cursor 确立了更有纪律的"AI 结对程序员"与"AI 原生 IDE"路线——这两条路才真正活了下来。

2023-03-14
OpenAI 发布 GPT-4 ——让多步自主智能体第一次变得可行，当年所有实验都建立在它之上。
2023-03
Cursor 首发 ——首个 AI 原生 IDE（VS Code 分支），把 AI 嵌进编辑循环，开创后来估值数百亿的品类。
2023-03-28
BabyAGI ——约 105 行的任务驱动智能体，被数千人 fork，普及了"规划-执行-再排序"模板。
2023-03-30

01 核心 Agent Loop（ReAct 工具循环） prompt → model → tool calls → observations → 重复；以及终止与轮次交接 +

是什么

每个编码智能体核心的那个小控制结构：持有一段不断增长的对话，问模型下一步做什么，执行它请求的工具，把结果追加回对话，然后重复，直到模型给出一轮没有工具调用的回复。它是 ReAct 的务实后代，但 2026 年的生产工具都建立在供应商原生的 function/tool-calling API 上，而非解析自由文本的 "Thought/Action/Observation"。

怎么工作

状态是一个有序的 messages 数组 + 静态前缀(system prompt + 每个工具的 JSON Schema)，整个数组每轮全量重发——因此不加缓存时 token 成本是二次方的。模型的一轮回复是若干内容块：text、thinking、以及请求工具的 tool_use 块。你必须在下一条 user 消息里，用 id 严格匹配地回答每一个 tool_use（tool_result）。最重要的细节是终止：循环在模型产出"零个 tool_use"的一轮时结束——"没有工具调用"就是完成信号，默认没有别的。只读工具(Read/Glob/Grep)可并行，改状态的工具(Edit/Write/Bash)串行执行以免冲突。

深入机制 · 线级协议、流式与终止

一次 assistant 回合是一个内容块列表。工具请求是一个 tool_use 块;你必须在下一条 user 消息里用逐一匹配的 tool_result 回答每一个——id 必须对齐、顺序与完整性都是承重的,漏一个或错配就是硬 API 报错。工具结果永远是 role:user("环境以用户身份说话")。

// assistant 回合(模型输出)
{ "role":"assistant", "content":[
  { "type":"text", "text":"我先看下这个文件" },
  { "type":"tool_use", "id":"toolu_01A…", "name":"read",
    "input":{ "path":"src/app.ts" } } ] }
// 你必须回的下一条(环境→模型)
{ "role":"user", "content":[
  { "type":"tool_result", "tool_use_id":"toolu_01A…",
    "content":"1\timport …", "is_error":false } ] }

终止由 stop_reason 驱动,是循环里最关键的分支:

end_turn模型给出纯文字、无工具调用 → 主终止,把控制权交回用户。
tool_use本回合请求了工具 → 执行、回填、继续循环。
max_tokens被截断在半途 → 不是完成!成熟循环发起续写请求,MVP 常误判为结束。
refusal / pause_turn模型拒答 / 长任务暂停 → 各自的处理路径。

Claude Code SDK 把这一切表面成一个类型化事件联合体:SystemMessage(init) → 每回合 AssistantMessage/UserMessage → 收尾 ResultMessage;ResultMessage.subtype 给出 success | error_max_turns | error_max_budget_usd | error_during_execution。Gemini CLI 的流暴露约 18 种事件类型。流式下工具输入以部分 JSON 增量到达,harness 必须缓冲到该 tool_use 的 input 完整才派发,否则发出畸形参数。

两种终止哲学:隐式(无工具调用=完成,Claude Code/Codex/Gemini)vs 显式终止工具(attempt_completion/finish,OpenHands/Cline/AutoGPT 血统)。后者给出可解析的停止点、可挂"最终答案 schema",但有"永不调用(死循环)"或"过早调用"风险。Gemini CLI 还额外跑一个 nextSpeakerChecker——每回合后用一次便宜的 flash-lite 调用判断"该继续还是交回用户",能抓过早停止,但翻倍往返、增加延迟,本身也是 loop bug 之源。

关键设计抉择

终止信号：隐式"无工具调用即完成"(Claude Code/Codex/Gemini) vs 强制模型调用显式 finish/attempt_completion 工具(OpenHands/Cline/AutoGPT 血统)。隐式更简单、让模型自然收尾；显式给出可解析的停止点，但有"永不调用(死循环)"或"过早调用"风险。
工具编码：原生 function-calling(结构化、支持并行、贴合训练，但限于强模型) vs 文本/XML/diff 解析(通用、省 token、容错，但串行且重试更多)。Cline v3.35 从 XML 迁到原生、保留 XML 兜底，是这个取舍的典型。
轮次仲裁：信任模型自己的收尾(便宜，一次往返) vs 额外跑一个 nextSpeakerChecker 分类器判断"继续还是交回用户"(Gemini CLI)——能抓住过早停止，但翻倍往返、增加延迟，本身也是死循环的来源。

常见陷阱

  • 漏答或错配 tool_use：每个 tool_use 必须有 id 匹配的 tool_result；报错也要返回 is_error 结果，绝不能什么都不返回。
  • 把 stop_reason:max_tokens 当成"完成"：模型是被截断了，成熟循环会发起续写请求。
  • 没有轮次/预算/时钟上限："改进这个代码库"这类开放请求会无限循环——务必设 max_turns + 花费预算 + 墙钟超时 + 连续失败熔断。
  • 把工具报错当致命错误让循环崩溃，而非当成模型可绕行的观察结果。
  • 来自观察结果的 prompt injection：工具输出(网页/文件/他人代码)是可被攻击者控制的数据，却以可信面貌重入上下文——把观察当数据、不当指令。
MVP · 一个周末 ~100 行

单个 messages 数组；while 循环调用供应商 + 少量 JSON-Schema 工具(read/write/bash/grep)；执行 tool_use、追加 tool_result、重复；模型无工具调用即终止；一个硬 max_turns 防跑飞。非流式、串行、无权限、无上下文管理——这确实能跑，是正确的起点。

成熟版 · Claude Code / Codex 级

流式 + 类型化事件联合体喂给 UI；增量解析缓冲工具输入；只读并行/写入串行；完整权限系统(allow/ask/deny + 模式) + PreToolUse/PostToolUse hooks；沙箱；prompt 缓存把成本变线性；自动压缩 + 子智能体卸载;健壮的终止处理(end_turn / max_tokens 续写 / refusal / error) + 轮次/预算/时钟上限 + 连续失败熔断;会话持久化;每次结果的成本核算;MCP。

02 工具系统（read/write/edit/bash/grep/fetch） 智能体的双手：难点不在 schema，而在工具周围的契约 +

是什么

一小组固定的、带 JSON Schema 的函数，模型调用它们来观察和改动代码库：读/写/改文件、跑 shell、搜代码(grep/glob)、抓网页。真正难的不是 schema，而是围绕它们的契约：编辑如何表示才能让随机性的模型可靠产出、执行前 harness 强制什么(权限/沙箱)、失败如何反馈让模型自纠错而非空转。循环质量——而非 schema——才是"可用智能体"与"烧 token 机器"的分水岭。

怎么工作 · 编辑是核心难题

LLM 无法可靠产出带行号的 unified diff(会数错行号)，而全文件重写既费 token 又诱发"偷懒"(输出 // ...其余不变...)。生态收敛到约 4 种表示：①SEARCH/REPLACE(字符串锚定，old_string 须唯一，Claude Code/Aider/Cline 的默认)；②全文件写入(最简单但费 token)；③简化 unified diff(去掉行号、模糊匹配，Aider udiff 让 GPT-4T 偷懒率降 3 倍)；④上下文锚定 patch 文法(OpenAI V4A/apply_patch，靠匹配周围行而非行号，但模型是专门训练过的，换非 OpenAI 模型就失灵)。落地时按"精确匹配 → 去尾空格 → 归一化换行 → 缩进容忍 → 阈值模糊"逐级回退，低于置信阈值就拒绝并报精确错误，绝不破坏性地猜。

深入机制 · 四种编辑表示与逐工具实现

① SEARCH/REPLACE(字符串锚定)模型给字面 old_string + new_string,harness 找唯一匹配替换,无行号。Claude Code 的 Edit、Anthropic str_replace_based_edit_tool、Aider "diff"、Cline replace_in_file、OpenHands str_replace_editor 都是它。安全默认。
② 全文件(write_to_file)模型吐整个新文件。零匹配失败,但费 token 且诱发"偷懒"(// ...unchanged...)。用于新建/整体重写。
③ 简化 unified diff(Aider udiff)去掉 hunk 头行号、靠上下文模糊匹配。Aider 实测让 GPT-4T 偷懒率降 3 倍(20%→61%),因为它促使模型重写整个函数。
④ 上下文锚定 patch 文法(V4A / apply_patch)OpenAI 的 *** Begin Patch / *** Update File / @@ 上下文 / -旧 / +新,一个信封含增删改移;模型被专门训练过——换非 OpenAI 模型就失灵。经 process_patch 走文件系统,不走 shell。

落地匹配机制:先试精确字节匹配,失败逐级升级(去尾空格 → 归一化 CRLF/LF → 缩进容忍 → 相似度模糊);OpenCode 据报串了 9 个匹配器;Cline 做了"顺序无关的多 diff 应用"让乱序块也能整批落地;然后算出新文件、(可选)展示 diff 供批准、原子写(temp+rename)、更新记录的 mtime。绝不静默应用低于阈值的模糊匹配。另有"apply model"变体(Cursor 式):聪明模型松散说"改什么",便宜专用模型把它变成精确字节。

其余工具的关键细节: bash 输出封顶(Claude Code 30k 字符、150k 天花板),溢出则把全量写盘、只回路径 + head 预览让模型 grep;支持 run_in_background;描述主动把模型从 find/grep/cat/sed 引导向专用工具。grep 由 ripgrep 支撑(尊重 .gitignore、全正则、output_mode 选 files/content/count),glob 按 mtime 排序(最近改的通常最相关)。web_fetch(url,prompt) 把 HTML 转 markdown 后交一个小快模型按 prompt 作答(巨页不进主上下文);HTTP 升 HTTPS、跨主机重定向返回给模型而非自动跟随(挡住重定向绕过白名单)、域名 allow/deny 在权限层用 WebFetch(domain:…) 强制。

错误串是模型唯一的感官反馈,是一等设计面:①具体(命名工具+失败类型+触发输入:"old_string 未找到" vs "匹配 3 处,加上下文");②可操作(追加"别用相同参数重试"在非正式测试里把相同重试从 ~60% 降到 ~10%);③清晰终止态(SUCCESS/FAILED);④原样回传编译/测试/lint 输出;⑤N 次相同失败后升级/退出(OpenCode 记录在案的无限编辑重试 bug 就是反例)。

关键设计抉择

编辑表示是最大的分叉：字符串锚定 SEARCH/REPLACE(安全默认，无行号运算) vs 全文件(最简但费 token) vs 训练过的 patch 文法 V4A(最可靠但只服务于训练它的那个模型)。除非你自己拥有模型，选 SEARCH/REPLACE。
匹配严格度：只精确匹配(可预测但脆，一个尾空格就失败) vs 模糊(救回更多但可能改错位置)。共识：逐级容忍回退，但低于阈值就带好错误信息拒绝。
权限 vs 沙箱是两道不重叠的闸：权限决定执行前"能不能跑"，OS 沙箱决定执行后"能碰什么"。两者都需要。

常见陷阱

  • 非唯一 old_string 悄悄改错位置——强制唯一或要求足够上下文，并为有意场景提供 replace_all。
  • 空白/缩进/换行(CRLF vs LF)不匹配是编辑失败头号原因；在回退里归一化，但绝不低于置信阈值。
  • 无限重试循环：编辑失败后模型永远重发相同参数烧 token(真实 bug)——用具体+可操作的错误 + "别用相同参数重试" + N 次后硬退出。
  • 无界输出摧毁上下文窗口(cat huge.log、5 万行 grep、巨大网页)——一律截断并溢出到文件，只回传路径 + 预览。
  • 用 bash 参数规则约束 curl URL 是脆的(复合命令可绕过)——改用 fetch 工具的域名规则。
MVP

五个工具：带行号的 read；全文件 write；精确+唯一匹配的 edit(不中就回"未找到 old_string")；带超时和输出上限的 bash；glob 或 bash-grep。写/跑前一律 y/n 确认。这基本就是 Anthropic text-editor + bash，已能搞定多数单文件任务。

成熟版

edit 变健壮：read-before-edit + mtime 陈旧检查、回退匹配级联、replace_all、顺序无关的批量 MultiEdit、写前给 diff 审阅、原子写入。读/跑/抓都截断-溢出到文件。bash 加后台执行、持久 shell、per-command 权限规则、继承式 OS 沙箱。搜索用专用 ripgrep Grep + 按 mtime 排序的 Glob。错误串工程化：具体、可操作、标注终止态、原样回传编译/测试输出、重复失败后升级。

03 上下文与记忆管理 把无限增长的对话塞进固定窗口，同时不丢关键信息 +

是什么

由于循环每步都把整段对话重发给模型，上下文只增不减：每次工具调用都追加调用本身 + 它(往往巨大)的结果。管理发生在四层：①预算与度量(在窗口里量了多少)、②压缩/摘要(满了怎么办)、③持久记忆文件(CLAUDE.md/AGENTS.md/GEMINI.md)与检索(每轮拉对的上下文进来)、④prompt 缓存(让由此产生的巨大稳定前缀重发起来又便宜又快)。

怎么工作 · 压缩的三个层级

层1 工具结果卸载(最便宜，无模型调用)：把臃肿的工具输出写到盘上，行内换成一个可再读的引用/路径——因为工具输出通常占了 token 大头，这一招就回收了大部分窗口。层2 就近截断：成对(user/assistant)地丢最旧的轮次以保结构。层3 LLM 摘要(full compact)：在 ~80-90% 时用专门 prompt 把 transcript 提炼成"目标/决策/改过的文件/待办/报错/下一步"，用摘要替换旧消息，再做 rehydration(重读近期文件、恢复待办、重新注入 CLAUDE.md)。OpenHands 把压缩记为事件流里的一个 CondensationAction，用重放算出视图，让压缩可审计、condenser 无状态。

深入机制 · 度量、Condenser 变体与事件溯源

正确的度量分母是"总窗口占用 = input + cache-creation + cache-read + output",不是只看 input_tokens(缓存 token 仍占窗口)。阈值触发时必须为"下一轮输出 + 压缩调用本身"预留缓冲。各家阈值:Claude Code ~80-95%(CLAUDE_AUTOCOMPACT_PCT_OVERRIDE 可调)、Codex model_auto_compact_token_limit(~180k-244k、只能调低)、Gemini CLI 早压缩 ~50%。

OpenHands 把压缩形式化为 Condenser(condense(events) → events),给出可插拔变体:

RecentEvents直接丢旧事件——便宜、确定,但忘掉早期决策。
LLMSummarizing(默认)keep_first N + 摘要中段 + 保留近期,语义保真但有一次模型调用、且有损。
StructuredSummary用 function-calling 填 ~18 个固定字段——可预测、机器可解析,但僵硬。
LLMAttention让模型按相关性给事件排序后取舍。

关键架构选择:把压缩记为 append-only 日志里的一个 CondensationAction 事件,视图 = 重放(日志 − 被遗忘 + 摘要),从而让 condenser 无状态、全量日志可审计可回退。压缩可在回合中途的工具循环边界触发(Codex),而不只在用户回合之间。Codex 对自家模型走 compact() API 返回不透明的 encrypted_content 密文(服务端解密无损恢复隐状态,但供应商锁定、不可移植);对开源模型则用本地摘要 prompt + handoff prompt。Cline 成对(user/assistant)删除并按目标窗口缩放力度(200k→64k 时删 ~75%),并区分 /smol(原地压缩保动量)与 /newtask(只打包 计划+文件+下一步 到干净窗口)。

压缩后的 rehydration 不可省:重读近期文件、恢复待办、重注入 CLAUDE.md,否则智能体在已改过的文件上按陈旧心智模型继续。记忆文件层级(enterprise>project>dir>user)每轮拼进 system prompt,@path 导入行内展开(是组织手段、不省上下文),祖先文件启动即全量加载、子目录文件按需加载(避免 monorepo 膨胀)。↳ prompt 缓存与结构化 repo-map 检索的完整机制,见下方「深入专题」的缓存与推理经济学、代码智能与检索两卡。

关键设计抉择

触发阈值：高(~90%+，最大化利用但风险是下一轮很大就没空间做摘要调用) vs 早(~50%，Gemini CLI，稳但浪费窗口、压缩更频繁)。必须为"输出 + 摘要调用本身"预留缓冲。
度量基准：事件/消息计数(简单但每事件 token 差异极大，仍会溢出) vs 含缓存 token 的真实 token 核算(准确，需要 tokenizer 或计数端点)。
缓存稳定 vs 新鲜：激进缓存要求前缀逐字节稳定，这与"会话中途改工具、切模型、重写前缀式压缩"冲突——每一个都会悄悄让缓存失效。方向性 partial-compact 正是为解决它而生。
检索策略：结构化(Aider repo map——tree-sitter + PageRank，便宜确定、无需索引基建) vs 语义 RAG(找概念相近，需索引 + 新鲜度处理) vs 纯 agentic grep/glob(无索引，模型驱动搜索，但费轮次)。

常见陷阱

  • 只用 input_tokens 对窗口计量——cache-creation/cache-read/待输出都占窗口，少算就会在"安全"读数下硬溢出。
  • 用消息/事件计数当 token 代理(OpenHands 记录在案的缺陷)：一次大文件读就能盖过 50 个小轮次。
  • 把 CLAUDE.md 当成"减少上下文"：@imports 会行内展开、全额计入窗口；臃肿的 CLAUDE.md 悄悄吃预算并降低指令遵从(保持 <200 行)。
  • 把记忆文件当强制约束：它们是模型可忽略的建议性上下文——硬约束要放进 PreToolUse hook，而非 Markdown 规则。
  • 靠压缩当长期记忆：会话内摘要随会话消失——要持久的东西必须写进记忆文件或外部存储。
MVP

用本地 tokenizer 或 chars/4 计数；超 ~70% 就成对就近截断；从仓库根加载一个 CLAUDE.md 原样拼进 system prompt；文件选择靠模型自己调 grep/glob/read；在 system prompt + 工具定义之后放一个缓存断点。既连贯又便宜，拿到 80% 的价值。

成熟版

含缓存 token 的精确核算 + 预留缓冲 + per-model 阈值；三层压缩(卸载→截断→摘要+rehydration) + 熔断；事件溯源历史(append-only 日志 + 计算视图)让压缩可审计可回退；分层惰性加载记忆(enterprise>project>dir>user)；结构化 repo map + 语义检索;缓存感知的前缀排序(最稳在前) + 方向性 partial 压缩 + TTL 管理;跨会话持久化;给用户的上下文用量指示器 + 手动 /compact /newtask。

04 权限、安全与沙箱 在模型的工具调用与真正执行之间：什么静默跑、什么问人、什么禁止 +

是什么

坐在"LLM 提议的工具调用"与"真正执行"之间的一层，决定什么静默运行、什么暂停问人、什么彻底禁止。它组合了策略引擎(对 工具+参数 模式的 allow/ask/deny 规则)、交互模式(plan/auto-accept/bypass)、以及 OS 或容器级沙箱(即便面对被攻陷或失控的智能体也能强制文件系统与网络边界)。目的是纵深防御:对安全操作降低批准疲劳,对破坏性/外泄操作硬性拦截——因为模型本身不能当最后一道防线。

怎么工作 · 两道分离的闸

成熟工具刻意区分两个正交控制:批准策略决定"harness 何时必须暂停问人",沙箱决定"命令运行时技术上能做什么"——它们可组合(一个调用可被策略自动放行,却仍被沙箱限制)。策略引擎把每个调用按有序规则匹配,Claude Code 的经典顺序是 deny → ask → allow 首个匹配生效,且匹配必须懂 shell(safe-cmd * 规则绝不能匹配 safe-cmd && rm -rf /)。沙箱层用 macOS Seatbelt、Linux bubblewrap + seccomp/Landlock,规则须在 execvp 之前装好,所有子进程继承——一个 npm postinstall 脚本和父进程一样被关进盒子。域名级网络控制则靠用户态代理白名单实现(OS 沙箱只懂 socket、不懂域名)。

深入机制 · 规则引擎、OS 沙箱原语与网络代理

规则求值:Claude Code 的经典顺序是 deny → ask → allow 首个匹配生效,且特异性不覆盖顺序——宽泛的 deny Bash(aws *) 击败更窄的 allow Bash(aws s3 ls);匹配到 ask 就触发,即便存在更具体的 allow。规则是 Tool 或 Tool(specifier) 模式(命令 glob 或路径 glob),如 Bash(npm run test:*)、Read(./.env)、WebFetch(domain:api.github.com)。匹配必须懂 shell:解析操作符而非朴素前缀,否则 &&/管道/$()/重定向轻易走私。配置分层合并 enterprise > user > project > local,任何来源的 deny 跨合并取胜——组织能锁死开发者无法放松的策略。

OS 沙箱原语(策略被绕过也守得住的那层):

macOS Seatbelt用 sandbox-exec 加一个生成的 .sb profile 限制文件系统/网络。
Linux bubblewrap建全新 mount+user+network 命名空间;配 seccomp-BPF 过滤系统调用(拦 connect/bind/listen、放行 AF_UNIX)和/或 Landlock 做写作用域。
装载时机规则须在 execvp 之前装好,子进程从第一条指令即受限;PR_SET_PDEATHSIG 让孤儿子进程随父而死。
继承性决定性优势:在内核/命名空间层强制,所有子进程继承——npm postinstall 与父进程一样被关进盒子。

网络域名过滤:OS 沙箱粒度粗(只懂 socket),所以去掉沙箱的网络命名空间、把全部流量经 Unix socket 强制到一个查白名单的用户态代理(对被拒 CONNECT 回 403、对新域名提示),再用 seccomp 兜底那些忽略代理环境变量的工具。无深包检测,只在域名/CONNECT 级过滤。隔离粒度:worktree-per-agent 只隔离文件(共享对象库、near-instant、硬链接),一旦智能体绑端口/碰 DB/缓存就需要 container-per-agent(Docker,或更强的 Firecracker/E2B microVM、gVisor/runsc)。OpenHands 默认就是 Docker client-server Runtime。逃生舱("带批准重试非沙箱")能大幅降摩擦,但正是被注入/过热心的智能体用来"说服你放它出盒子"的杠杆——真实案例里智能体找到 /proc/self/root/usr/bin/npx 后试图论证关掉沙箱。↳ 拦截点之上的可编程 hooks(PreToolUse 等)与 checkpoints,见下方「深入专题」的会话状态 / hooks / 可观测性卡。

关键设计抉择

两轴分离 vs 一个旋钮：把"何时问"(策略)与"能做什么"(沙箱)分开,才能实现"全部自动放行,但关在紧盒子里"这一推荐的自主配置。合并成一个 YOLO 开关更简单,却逼你做全有全无的安全选择。
强制位置：prompt 级(最便宜,模型可无视——plan mode 会漏) → harness/应用级(对自家工具可靠,但被子进程绕过) → OS/内核级(子进程继承,唯一能扛住 shell 出来的二进制)。成熟工具三层叠加。
逃生舱设计：允许"带批准重试非沙箱"能大幅降摩擦,但正是被 prompt 注入或过度热心的智能体用来"说服你放它出盒子"的杠杆(真实野外案例)。关掉它(allowUnsandboxedCommands:false)以自主换安全。

常见陷阱

  • 信任模型自我克制:prompt 级"只读/请先问"会漏——智能体会中途掉出 plan mode。强制必须在 harness/内核,而非 prompt。
  • 应用级检查被子进程绕过:只拦自家工具,一旦智能体 shell 出去(npm postinstall、spawn 的二进制)就不受限——除非在内核/命名空间层强制。
  • 朴素参数匹配被 shell 操作符(&& | $() 重定向、/proc/self/root/... 别名)轻易击穿——要解析 shell,别字符串前缀匹配。
  • 沙箱初始化失败时默认"软降级"为非沙箱执行只给个警告(真实脚枪)——安全场景要 failIfUnavailable。
  • 读也是外泄向量:能 Read secret 就把它拉进上下文(可能再经允许的网络调用出去)——在 Read 工具层 deny secret 路径,而非只防写。
MVP

单个执行前 hook 把每个调用分类 allow/ask/deny(对 工具名+首参 做 glob 匹配);'ask' 给 y/n/always 提示;硬编码 deny 列表(rm -rf、force-push、CWD 外写、.env 读);会话内 allowlist 让 'always' 生效。真隔离直接把整个智能体跑在一次性 Docker 容器或 git worktree 里,信任那个边界——很多团队实际就这么干(容器里跑 YOLO)。Aider 的 git 自动提交更是最便宜的 MVP:不设闸,但每次改动都可秒回滚。

成熟版

懂 shell 的参数解析封堵操作符走私;分层配置 enterprise>user>project>local + 组织不可覆盖的 deny;策略与沙箱显式分离;子进程继承的 OS 强制(Seatbelt + bwrap/seccomp/Landlock,Windows 走 WSL2);域名白名单出口代理(默认拒网) + seccomp 兜底;真正的 tool-gating plan mode;受保护路径即便 bypass 也拦;硬化开关 + 受审计的人工逃生舱;每智能体 worktree+容器隔离(隔离端口/DB/缓存,不只文件);checkpoints 回滚;可编程组织策略 hooks。

05 规划与任务分解 / 多智能体 todo 列表、plan mode、spec-first、子智能体编排——对抗目标漂移与上下文耗尽 +

是什么

把模糊的人类请求变成有序、可追踪的可执行步骤;当任务大到一个上下文窗口/一个智能体装不下时,把它拆到多阶段、多文件或多个协作智能体。四种由轻到重的机制:①模型写给自己的会话内 todo 列表;②把写操作挡在人工批准后的只读 plan mode;③把需求/设计/任务持久化为版本化产物的 spec-first 流;④把工作扇出到隔离上下文的子/多智能体编排。共同目标是对抗长跨度编码的两大失败模式:目标漂移与上下文耗尽。

怎么工作

这些几乎都是"普通工具 + system prompt 纪律",而非模型特殊能力。Todo 列表(TodoWrite/update_plan/write_todos)每次调用传入整个列表并覆盖(幂等替换,不是追加),没有引擎"执行"它——它是个自我提示脚手架,价值在于每轮重新注入列表把目标钉在上下文里。Plan mode 实现为一个模式标志,换掉工具白名单(只留只读)与 system prompt——写工具从工具集里物理消失,模型想改也改不了;回到执行本身是一个 exit_plan 工具,展示计划并阻塞到人工批准。子智能体:一个 Task/delegate 工具生成一个带干净上下文、自己的 system prompt 和(常常)自己模型的全新智能体,跑完只回一段文字摘要、其上下文被丢弃——这就是核心的上下文隔离技巧。

深入机制 · 四种由轻到重的实现

Todo 列表(TodoWrite / update_plan / write_todos)单次调用传入整个列表并覆盖(幂等替换,非追加),项为 {content, status: pending|in_progress|completed}。没有引擎"执行"它——纯自我提示脚手架;每轮重注入把目标钉在上下文里、给用户进度视图。系统 prompt 规则驱动:"3+ 步才用""恰好一个 in_progress""完成即刻标记""测试没过绝不标完成"。
Plan mode(工具集切换)一个模式标志换掉工具白名单(只留 Read/Grep/Glob/WebFetch)与 system prompt——写工具从工具集物理消失。回到执行是一个 exit_plan(plan_text) 工具:展示计划、阻塞到人工批准,批准才解锁写工具。Cline 把模式存在 StateManager 单例并广播给 webview,可 per-mode 加载不同模型+prompt。
Spec-first(盘上产物流水线)GitHub Spec Kit:/speckit.specify → .plan → .tasks → .implement(+ constitution/analyze);AWS Kiro:requirements.md → design.md → tasks.md 带逐阶段批准闸 + 保存即跑测试/lint 的 Hooks。计划外置到文件,挺过会话/上下文重置,能像代码一样 diff 审查。
子/多智能体委派一个 Task/delegate 工具生成带干净上下文、独立 prompt、(常)独立模型的子体;OpenHands 实现为一等 AgentDelegateAction 事件(父 AgentController 起子 controller);Claude Code 的 Agent Teams(实验)更进一步:共享任务表 + 状态/依赖跟踪 + 点对点消息 + 文件锁,每个队友是自己 tmux 面板里的完整 Claude 实例。

Aider 的 architect/editor 分离是按"模型角色"而非仓库访问来拆:架构模型用散文推理改动,再把文本交给一个只负责用严格编辑格式吐 diff 的、更便宜的 editor 模型——实测既提分又省 30-50% 成本(费 token 的 diff 生成跑在便宜模型上)。Codex 的 PLANS.md 模式(带 Progress / Surprises / Decision Log / Retrospective 段)支撑过单 prompt 7+ 小时的运行。

关键设计抉择

强制 vs 建议：plan mode 可以是硬闸(写工具从工具集移除——Claude Code/Cline)或软约定(prompt 说"先规划"——Codex 一两句计划)。硬闸防跑飞但有摩擦。
计划住哪：在上下文里(todo)便宜但随会话消失;在盘上(spec 文件)能扛住上下文重置、支持多小时/多会话,但有仪式感且可能与代码漂移。
同模型双角色 vs 分角色：Aider 证明把推理(architect)与编辑(editor)分开既提分又省 30-50% 成本(费 token 的 diff 生成跑在便宜模型上);代价是两次往返、延迟更高。
子智能体无状态 vs 共享状态：无状态(返回即死)可预测、防上下文膨胀但不能协作;共享任务列表+消息(Agent Teams)能真协作但加锁竞争与可观测性缺口。

常见陷阱

  • 目标丢失:约 10 轮编辑后模型忘了最初目标——这正是 todo/plan 每轮重注入的全部理由,省了它就是静默的默认失败。
  • plan mode 名不副实:若写工具还在工具集里、只靠 prompt 说"别编辑",模型终究会编辑——靠移除工具强制,别靠嘴求。
  • 过早标记完成:测试还没过就把任务标 completed——需要显式 prompt 规则(验证前保持 in_progress),工具本身不强制任何东西。
  • 子-子智能体无深度上限递归,成本与时间爆炸——每个子智能体 prompt 里放硬终止条件 + 编排器里限深度。
  • 无验证层:并行 worker 返回貌似合理却冲突的输出,编排器盲目拼接——一个交叉核对/对账 pass 才让扇出可靠而非只是快。
MVP

一个 update_todos(items[]) 覆盖会话列表({text,status});每轮渲染并重注入。加 ~5 行 system prompt:"3+ 步任务先写 todo;恰好一个 in_progress;验证后才标 completed"。这就抓住了 80% 的防漂移价值。可选:一个布尔 plan-mode 标志把工具集过滤为只读 + 一个 exit_plan 工具等 y/n。无子智能体、无文件、无依赖图。

成熟版

严格单-in_progress 不变量 + 状态完整性绑测试结果 + TUI 实时渲染;真模式系统(plan 换工具集 AND system prompt + 硬批准闸 + per-mode 模型/architect-editor 分离);spec-first 持久化(requirements→design→tasks + 逐阶段批准 + 一致性分析);子智能体编排(隔离上下文子体 + manager-worker + 依赖分析并行 + 递归深度上限 + 对账 pass);前沿:共享任务列表 + 依赖跟踪 + 点对点消息 + 文件锁(Agent Teams) + 每动作后自动跑测试/lint 的 hooks + 每智能体可观测性。

06 System Prompt 与 Harness 工程 把无状态聊天模型变成智能体的那层软件——"harness 才是产品" +

是什么

"harness"是 LLM API 调用周围的全部代码:组装 prompt(系统指令 + 工具 schema + 对话历史)、跑 reason→act→observe 循环、派发工具调用、把结果喂回、管理上下文增长、拦截危险动作、决定何时停。模型提供推理与选择下一步;harness 提供原始模型做不到的一切——持久化、记忆、碰文件/shell/网的能力、安全与迭代。2026 年 3 月 Claude Code 源码泄露后行业共识很直白:harness、而非模型,才是产品。

怎么工作 · System Prompt 是分层组装的

system prompt 不是一根字符串,而是由稳定段与易变段组合而成,排序刻意让稳定前缀能被 prompt 缓存(重复前缀省 ~90% token)。典型 Claude Code 分层:(a) 身份/人格 → (b) 行为规则(简洁输出、别擅自加注释、别擅自提交、prompt 注入护栏) → (c) 工具使用策略(用 Read 别用 cat、批量并行、一次一个 in_progress) → (d) 环境块(cwd/OS/git 状态/日期——易变,放后面减少缓存失效) → (e) 项目记忆(CLAUDE.md) → (f) 会话状态。工具描述本身就是 prompt 工程:何时用、何时别用、前置条件、示例,都是承载行为契约的指令文本。流式经 SSE 增量到达,工具参数片段要累积到完整 JSON 才派发;AbortSignal 贯穿整条调用以支持中断,计数器/锁要在 finally 里释放。

深入机制 · Prompt 分层、工具描述与模型路由

System prompt 是分层组装的,排序刻意让稳定前缀能被缓存(见下方缓存卡)。典型 Claude Code 分层,从最稳到最易变:

(a) 身份/人格"You are Claude Code…"。
(b) 行为规则简洁输出、别擅自加注释/提交、git 安全、prompt 注入护栏(继续前先标记可疑注入)。
(c) 工具使用策略用 Read 别用 cat、批量并行、一次一个 in_progress。
(d) 环境块cwd/OS/git 状态/日期——易变,放后面以最小化缓存失效。
(e) 项目记忆CLAUDE.md / AGENTS.md / GEMINI.md 拼入。
(f) 会话状态todo 列表、当前模式等。

工具描述本身就是 prompt 工程:每个工具 = JSON-schema 函数 + 一段自然语言描述,那段描述实为指令文本,承载行为契约——"用 Grep 别用 bash grep""编辑前必须先 Read""apply_patch 后别再读该文件——若没应用成功这次调用会报错"。两种派发风格:原生 function-calling(供应商把工具调用解析成结构化对象)vs 自解析 XML/文本(Cline 旧变体、Aider 的 SEARCH/REPLACE 编辑块)——后者能跑没有原生工具支持的模型,但要自写解析器 + 修复逻辑(把 Read 纠成 read、把畸形调用路由到一个 'invalid' 工具让模型自纠)。

模型路由两种模式:①按角色——强推理模型规划、便宜/快模型执行或编辑(Aider architect/editor、Cline Plan/Act);②按子任务——大模型驱动主循环,小模型(Haiku)干机械活:摘要/压缩、子体搜索、commit 消息。Aider 还按模型强弱匹配编辑格式(强→diff、弱→全文件)。Claude Code 的硬核细节(经 2026-03 源码泄露公开):约 2592 行 / 23 项编号检查的 bash 安全校验器、写死的 prompt 注入护栏、反蒸馏防御(注入假工具定义、摘要化推理)。OpenHands 把 agent 建模为"从不可变事件流到下一事件的纯函数",使暂停/恢复/fork/确定性重放/审计成为可能。↳ 可观测性/tracing 与事件溯源的完整机制,见「深入专题」的会话状态 / hooks / 可观测性卡。

关键设计抉择

原生 function-calling vs 自解析 XML/文本：原生健壮、供应商维护但锁定支持它的模型;XML/文本(Cline 变体、Aider 编辑块)能跑任意模型但要自写解析器 + 工具名/参数修复。
单智能体循环 vs 多智能体编排：Claude Code/Codex/Gemini 刻意保持一个推理循环 + 扁平消息列表以求可预测可调试;多智能体加能力也加协调成本与非确定性。
prompt 缓存布局 vs 新鲜：最大化稳定缓存前缀,与"保持上下文新鲜(git 状态/时间/文件态每轮变)"冲突——靠稳定→易变排序、只在尾部接受缓存失效来化解。
往 system prompt 塞多少 vs 用工具教：啰嗦行为规则提升遵从但每轮费 token,且超 ~25k token 上下文模型会"分心";精简 prompt 更便宜但更难控。

常见陷阱

  • 经工具结果的 prompt 注入:文件/网页/命令输出是与指令同处一个上下文的不可信输入;恶意仓库可劫持智能体——加显式护栏,把抓取内容当数据绝不当指令。
  • 缓存抖动:把易变数据(时间戳、git 状态、token 数)放前面,每轮都让 prompt 缓存失效、悄悄翻倍成本——按稳定→易变排序。
  • 流式部分标记泄露:逐 token 冲刷会把半成形的工具标签渲染成可见文本;参数没累积完就派发会发出畸形调用——缓冲到单元完整再动。
  • 缺中断清理:客户端会中途断开,若只在正常路径释放锁/计数器就会泄漏——在 finally 里释放。
  • 过度臃肿的工具集/描述:工具太多或"何时用"含糊会导致选错工具、浪费调用——精挑一小组、描述锋利。
MVP

几百行、一个周末:while 循环调用原生工具模型 + 一段短静态 system prompt + 3-4 个工具;执行工具调用、追加结果到扁平消息列表、循环到无工具调用;硬轮次上限(~25);可选写/跑前单个 y/N;全文件编辑;不流式;不做上下文管理;直接跑在宿主机。对小而清晰的任务确实能跑,是正确起点。

成熟版

模块化、缓存边界化的 system prompt(稳定→易变);富工具描述(何时/何时不/示例/前置);SSE 流式 + 增量参数累积 + 部分标记缓冲 + AbortSignal 中断 + finally 清理;按模型强弱匹配编辑格式 + 应用修复;模型路由(planner vs editor + 便宜模型做摘要/子体);反应式压缩(或主动 repo-map 选择) + 持久记忆重注入;真权限引擎 + 深度命令校验;沙箱执行;工具结果上的注入检测;MCP;多重停止条件;可观测性(记录每个 action/observation 以便重放与评估——OpenHands 视其为最重要的设计决策)。两者的差距几乎全是 harness 工程,而非模型质量。

MCP MCP 协议 · 智能体的可扩展插件接口 host/client/server + JSON-RPC 2.0，让一个 client 消费上千个 server、一个 server 通吃所有宿主 +

是什么

MCP(Anthropic 2024-11 提出)是一个开放的 JSON-RPC 2.0 协议,标准化"LLM 宿主如何连接外部工具、数据与 prompt",让编码智能体有一个统一插件面,而非为每个集成写胶水。一个 MCP client 实现就能消费任意社区/企业 server(GitHub、Postgres、Sentry、浏览器);反过来,一个 server 在 Claude Code / Cursor / Cline / Codex / Gemini CLI / Zed 之间零改动通用。

深入机制 · 拓扑、传输、生命周期与工具注入

拓扑:一个 HOST 进程(智能体应用)为每个 SERVER 起一个 CLIENT,各 client 与其 server 保持有状态 1:1 会话;host 把所有 server 的工具聚合进模型的工具 schema。传输:①stdio——server 作子进程,stdin/stdout 上换行分隔 JSON,本地默认、延迟最低;②Streamable HTTP(2025-03 起)——单端点 POST(客户端→服务端)+ 可选 GET 开 SSE 流,Mcp-Session-Id 头管会话,必须校验 Origin(不符回 403)防 DNS rebinding;③旧的双端点 HTTP+SSE 已废弃。

// 1) 握手：能力协商
→ initialize { protocolVersion, capabilities, clientInfo }
← { protocolVersion, capabilities, serverInfo, instructions? }
→ notifications/initialized
// 2) 发现 → 注入模型 schema（命名空间化）
→ tools/list  ← { tools:[{ name, description, inputSchema(JSON-Schema), outputSchema? }] }
   host 把每个 tool 译成 provider 原生函数 schema，名如 mcp__<server>__<tool>
// 3) 模型发起调用 → host 回映射
→ tools/call { name, arguments }
← { content:[…], structuredContent?, isError }   // 工具错误 in-band 用 isError:true（让模型自纠），不是 JSON-RPC error

三个 server 原语:tools(模型控制的函数)、resources(URI 寻址的只读数据,用于注入上下文)、prompts(用户可选模板,常呈现为斜杠命令)。三个 client 原语:sampling(server 反过来请 host 跑一次 LLM 补全 → 嵌套智能体循环)、roots(把工作区目录边界告诉 server)、elicitation(2025-06 起,server 用 JSON schema 请求结构化用户输入,host 渲染表单)。规范演进:2025-06-18 加结构化输出(outputSchema)、elicitation、OAuth 2.1 资源服务器(RFC 8707 资源指示符绑定 token)、移除批处理;2025-11-25 加实验性异步 Tasks(task{ttl} → 立即返回 taskId,轮询 tasks/get/tasks/result)、OAuth CIMD/OIDC、图标、JSON Schema 2020-12 为默认方言。

关键设计权衡

stdio vs Streamable HTTP：stdio 最简、本地延迟最低但一进程一 server、无网络;HTTP 可远程/多客户端并支持 OAuth,但要 Origin 校验、会话管理与鉴权复杂度。
工具命名空间与数量：聚合太多 server 会撑爆模型工具表(涨 token、选错工具)——必须命名空间化(mcp__server__tool),并常做 gate/allowlist 或懒加载压小 schema。
同步 tools/call vs 异步 Tasks：阻塞调用简单但长任务卡住循环;Tasks 释放循环但要轮询、TTL/清理与 task-ID 访问控制。
信任 server 文本：工具描述与结果进入模型上下文,当作可信 vs 不可信,决定了对工具投毒与间接注入的暴露面。

常见陷阱（安全面尤其重要）

  • 工具投毒(tool poisoning):server 在工具描述/schema 里埋对用户不可见、但被模型读到的恶意指令,劫持智能体外泄数据或跑命令——描述是不可信输入,不是文档。
  • Rug pull:一次批准的工具日后悄悄改定义;宿主缓存了旧批准,新会话继承被投毒的定义——需版本钉定、描述哈希/签名、变更后重新批准。
  • 经工具结果的间接注入:返回内容(网页、issue 正文、DB 行)含对抗指令被模型照做——把结果当数据、沙箱化副作用、在 MCP 层校验 I/O。
  • 混淆代理 / token 透传:持有宽 OAuth 凭证的 server 被诱导替不该有权限的调用者行事——2025-06 用 RFC 8707 资源指示符把 token 绑定到特定 server 修复。
  • DNS rebinding:不校验 Origin 或绑到 0.0.0.0,恶意网页就能打到 localhost MCP server——必须校验 Origin(403)并绑 localhost。

是什么

智能体循环每轮重发一段不断增长的 transcript(system + 工具定义 + 全部历史工具调用/结果 + 新输入),而 LLM API 是无状态的——不做服务端前缀缓存时,N 轮的累计 prefill 成本是 O(N²) token。前缀缓存把重复前缀变成 ~0.1x 的 cache-read 而非全价重算,把循环累计成本从二次方压到近线性。理解缓存键怎么算(逐字节前缀匹配)、什么会悄悄失效、以及 thinking token / tool_choice / 约束解码如何交互,是构建时对成本与延迟最大的单一杠杆。

深入机制 · KV 缓存、失效层级与约束解码

机制:Transformer 的 prefill 为每个输入 token 算出每层的 key/value 张量("KV 缓存"),这是请求里最贵的计算密集部分;解码输出则是带宽密集、相对便宜。前缀缓存把这些 KV 张量按 token 序列的滚动哈希(通常按 ~128-token 页量化)持久化,后续共享前缀的请求复用已存 KV、只 prefill 新增后缀。为什么 O(n²):第 i 轮重发 ~c·i 的前缀,∑ = O(N²);有缓存后每轮只重算 ~常量后缀 + 对缓存前缀付 0.1x,累计变线性。

Anthropic(显式断点):用 cache_control:{type:"ephemeral"} 标记边界,每请求最多 4 个;渲染/哈希顺序固定 tools → system → messages,断点缓存其之前的一切。定价:cache-read ~0.1x 输入价,cache-write 1.25x(5 分钟 TTL)/ 2x(1 小时 TTL)。回本点:5 分钟 TTL 第 2 次请求即回本,1 小时约需 3 次。最小可缓存前缀因模型而异(Sonnet 4.5 = 1024,Opus 4.x = 4096 token),不足则静默不缓存(cache_creation_input_tokens=0、无报错)。失效是前缀精确的,且分层:

改工具定义 / 换模型炸掉全部三层(tools+system+messages)。
改 system 内容炸 system+messages,保住 tools。
改 tool_choice / 开关 thinking / 图片变化只炸 messages,保住 tools+system。
20 块回溯窗口断点最多回溯 ~20 个内容块找旧条目;长回合(许多 tool_use/tool_result 对)会超窗静默 miss,需每 ~15 块插一个中间断点。

OpenAI/Gemini(自动前缀缓存):无断点,自动缓存最长已见前缀(≥1024 token、128 增量),无写入溢价、无额外费,折扣 50%(旧)~90%(新)。Thinking/推理 token:扩展思考产出的 thinking 块按输出 token 计费;同模型续接时必须原样回传 thinking 块(签名校验,改动会 400),换模型则丢弃(不计费);OpenAI o 系列推理是加密不透明的 encrypted_content。thinking 块位于 messages 层,开关它只炸 messages 缓存。约束解码(constrained decoding):strict:true 或 output_config.format 把 JSON Schema 编译成 token 级语法/FSM,解码每步屏蔽会违反语法的 token,保证语法有效、合 schema 的输出——这就是工具参数"保证有效"而非"祈祷有效"的原理;首次用某 schema 付一次编译延迟,之后 24 小时 schema 缓存。

核心概念

KV 缓存prefill 产出的每层每 token 的 key/value 张量;前缀缓存持久化它以跳过重算——请求里最贵的部分。
逐字节前缀匹配缓存键取渲染后 prompt 到断点为止的精确字节;前缀里任何一字节不同,就让该位置起的一切失效——一切规则的根源。
cache-read vs cache-write读 ~0.1x、写 1.25x(5min)/2x(1h);OpenAI/Gemini 自动缓存无写入溢价。
分层失效tools/system/messages 三层;改工具/换模型炸全部,改 system 炸后两层,改 tool_choice/thinking 只炸 messages。
推理/thinking token按输出计费;Anthropic 同模型须原样回传(签名校验),OpenAI 返回加密不透明推理。
parallel_tool_calls一回合可多个 tool_use;所有 tool_result 必须在单条 user 消息里回,拆开会悄悄训练模型不再并行。

关键设计权衡

断点位置：把 cache_control 放在稳定共享前缀的末尾,而非整个 prompt 末尾——标在多变尾部会让每个请求都写一个永不被读的新条目(白付 1.25x 写入)。
冻结 system prompt：绝不把当前日期/用户 id/模式插进 system prompt(它在前缀 0 位,会炸掉一切);把易变上下文作为靠后的消息注入。
工具集稳定 vs 动态：工具在 0 位渲染,增删/重排任一工具炸掉整缓存;大/动态工具库用 tool-search(追加 schema 而非替换)保住 0 位缓存。
循环中途换模型：缓存按模型隔离,主循环换便宜模型会炸缓存——应为便宜模型开独立上下文的子智能体,而非切主循环的模型。

常见陷阱

  • system prompt 或工具定义里的一个 datetime.now()/UUID/未排序 json.dumps() 让每请求都炸缓存——diff 两次请求的渲染字节来定位。
  • 断点标在含"本次问题"的整 prompt 末尾:每请求写新条目、无一被读,白付写入溢价。
  • 前缀低于模型最小值(如 Opus 4.8 的 4096 floor 下的 3K)静默永不缓存,同 prompt 在 Sonnet 4.5 的 1024 floor 却能缓存,无报错。
  • 把并行 tool_result 拆到多条 user 消息,悄悄训练模型停止并行,拖垮整会话吞吐。
  • N 个同前缀请求启动时齐发:缓存要等首个响应开始流式才可读,于是 N 个全付全价——先发一个、等首 token,再扇出。

深入机制 · Aider repo map 的完整算法

解析:每个文件按扩展名选 tree-sitter 语法,用一段 tags.scm S-表达式查询捕获 name.definition.* 与 name.reference.* 节点,产出 Tag(file, name, kind, line);缺 tags 查询的语言退回 Pygments 词法器(仅 references)。结果按文件 mtime 缓存在 .aider.tags.cache.v4(diskcache)。建图 + 排名:建一个 networkx 有向多重图,节点是文件;对每个标识符,从引用它的文件向定义它的文件连边,边权 = use_mul · sqrt(num_refs)。乘子叠加:

# 边权乘子（都会相乘叠加）
ident in mentioned_idents      → mul ×= 10   # 用户/对话提到的标识符
well-named ident（≥8 字符 驼峰/蛇形） → mul ×= 10
leading underscore（私有 _x）   → mul ×= 0.1
ident 有 ≥5 处定义（太常见）    → mul ×= 0.1
referencer 是 chat 文件         → use_mul ×= 50
# PageRank：personalization 向量把中心性偏向"当前任务在意的东西"
ranked = pagerank(G, weight="weight", personalization=…)  # chat/提到的文件/路径匹配加权

PageRank 得分再分摊回该文件的出边以排名单个定义,得到全局排序的 (文件, 符号) 列表。塞进预算:因为渲染后的 token 数与 tag 数非线性,做二分搜索——起点 middle = min(map_tokens//25, num_tags),渲染计数,落在 --map-tokens(默认 1024;无 chat 文件时 ×8)的 15% 内就接受,否则二分。渲染用 grep_ast.TreeContext:只打印排名靠前符号的签名行、其余折叠成 ...,于是模型看到的是骨架而非全文。对比:LSP(opencode 自动装 40+ 语言服务器)经 JSON-RPC 把 textDocument/publishDiagnostics 喂回循环,编辑后重读诊断作正确性闸,并可用 definition/references/hover 做编译器级精确导航;嵌入 RAG(Cursor)客户端建 Merkle 树、只重嵌变更叶子(~10 分钟一次)、按 AST 边界分块、路径用 6 字节确定性 nonce 加密后存 Turbopuffer,服务端不落明文;纯 agentic(Claude Code)无索引,靠成本递增的工具集(Glob<Grep<Read<Explore 子体)由模型自己迭代,靠 ~92% 前缀缓存复用维持经济性。

核心概念

Repo map按 token 预算裁剪的、全仓排名骨架(顶部符号显示为签名、body 省略),作环境上下文注入,让模型不读文件也知道"有什么"。
tags.scm 查询每语言一份 tree-sitter S-表达式,把节点标为定义/引用——让 parse 树对排名器有意义的唯一东西。
个性化 PageRank被 personalization 向量偏向 chat/提到的文件与标识符的图中心性,让排名反映当前任务而非全局重要性。
token 预算二分因渲染 token 与 tag 数非线性,二分包含的 tag 数直到落在预算 15% 内。
LSP-as-tool / 诊断闸跑真语言服务器、把编译器诊断喂回模型——精确的地面真相而非统计猜测,并作编辑后正确性闸。
Merkle 增量索引Cursor 把文件哈希成 Merkle 树、只重嵌变更叶子,在大仓上界定重索引成本。

关键设计权衡

有索引 vs 无索引：索引(repo map/嵌入)给即时广召回,但要建、要存、要保鲜;纯 agentic 搜索永远最新、零维护,但每次查询付延迟/token,且会漏掉"叫不出名字"的东西。
词法(grep) vs 语义(嵌入)：grep 精确命中确切符号但对概念查询("鉴权在哪")失效;嵌入相反,并引入分块/陈旧/索引复杂度。Augment 靠 grep+find 登顶 SWE-bench——"grep 与 find 就够了"。
算力花在哪：离线预算(嵌入/Merkle 同步)拿隐私+基建换延迟;查询时算(agentic)保数据在本地,但把成本摊进每次会话。
降级画像：repo map/嵌入在命名糟糕的巨型 monorepo 上退化;LSP 在无对应语言服务器时退化;agentic 在陌生代码库(模型还叫不出目标名)时退化——真系统通常分层叠加。

常见陷阱

  • 嵌入陈旧:两次重索引之间向量库描述的是旧代码,智能体对着幻影版本编辑——这是终端智能体弃嵌入转 live grep 的头号原因。
  • 分块破坏上下文:固定窗口把函数拦腰截断或把定义与其 docstring/调用者分开,检索块各自不连贯;AST 感知分块可缓解但加解析复杂度。
  • 语义匹配确切符号失灵:嵌入返回"看起来相关"的代码,而编码要精确符号——模糊匹配注入貌似合理却错误的上下文,模型随后信了它。
  • repo map 过度省略:TreeContext 只显签名,可能藏掉那一行关键实现,模型自信地在缺了关键的骨架上推理。
  • LSP 运营脆弱:服务器崩溃、冷启动慢、项目根错、或某语言无服务器,让智能体对诊断静默失明——且许多栈只接了诊断、没接导航,悄悄退回 grep。

是什么

深入机制 · 四层接口

① ACI(Agent-Computer Interface,SWE-agent,arXiv 2405.15793):ACI 是 LLM 看到的命令集 + 反馈格式。具体设计:有状态文件查看器只显~100 行/回合(论文实测最优)配 open/scroll/goto 跨回合保持光标,替代会灌爆上下文的 cat;edit <start>:<end> 替换行区间并先跑语法 linter,语法错的编辑被拒并回错(智能体无法把文件改坏);search_dir 只回 文件名+匹配数(带上下文反而让模型困惑);空 stdout 换成显式"命令成功、无输出";输出有界。原则:动作简洁且以 LM 为中心、把多步合并成单命令、护栏防错误态、反馈信息丰富而简洁。结果:SWE-bench pass@1 12.5%(当时 SOTA)。

② ACP(Agent Client Protocol,Zed,2025-08):JSON-RPC 2.0 over stdio——编辑器(客户端)把智能体作子进程起。握手 initialize(协议版本 + 客户端能力如 fs/terminal 支持)→ 智能体回能力 + 鉴权方式;session/new(cwd + MCP server 列表)返回 sessionId,session/load 恢复;session/prompt 发用户回合。工作时智能体推 session/update 通知(agent_message_chunk / thought / tool_call 及其 pending→in_progress→completed 状态 / plan / mode)。副作用则回调进客户端:session/request_permission(编辑器弹 allow/reject)、fs/read_text_file/fs/write_text_file、terminal/*——智能体不直接碰盘,编辑器对工作区与权限保持权威。刻意复用 MCP 的内容块类型。常被称"AI 编码智能体的 LSP"。

③ A2A(Agent2Agent,Google 2025-04 → Linux 基金会 2025-06):对等智能体经 /.well-known/agent-card.json 的 Agent Card 互相发现(name/url/skills/capabilities/securitySchemes/transport)。三种等价传输(JSON-RPC over HTTP / gRPC / REST);核心方法 message/send、message/stream(SSE)、tasks/get|list|cancel。一个 Task 是有状态、可长跑的单元,生命周期 submitted→working→(input-required|auth-required)→completed|failed|canceled|rejected。智能体是"不透明"的——只暴露 skills,不暴露内部工具/状态。A2A vs MCP:MCP 连"智能体→工具"(动词式函数、同步);A2A 连"智能体→对等智能体"(不透明、有状态、异步任务委派)——互补而非竞争,一个 A2A 智能体内部可以用 MCP。④ AG-UI(CopilotKit):事件式"智能体→前端",HTTP POST + SSE 约 16 种事件(TEXT_MESSAGE_*、TOOL_CALL_*、STATE_SNAPSHOT、STATE_DELTA 用 JSON-Patch),支撑 token 流、共享可变状态、人在环内、生成式 UI。

有界/信息丰富的反馈ACI 输出截断、空结果显式化、搜索只回文件名——每 token 高信号,模型不被淹没也不瞎猜。

关键设计权衡

深入机制 · 事件溯源 / shadow git / hooks 双通道 / OTel

① 事件溯源(OpenHands V1):对话是不可变 Event 的 append-only 日志,每个带 id/timestamp/source∈{user,agent,environment}。两大族:ActionEvent(智能体决定的工具调用,携 thought/thinking_blocks/tool+args/security_risk/llm_response_id)与 ObservationEvent(结果),外加 UserReject/AgentError/CondensationSummary。UI、智能体、运行时互不调用,只向日志追加/读取,于是每次运行"构造上即可重放"。LLM prompt(称"View")是算出来的、不是存下来的:过滤出 LLMConvertibleEvent、逐个 to_llm_message()、把共享 llm_response_id 的并行 Action 合成一条 assistant 消息、再过 condenser。恢复 = 重放日志;fork = 复制 [0..N] 事件到新 conversation_id 续跑(因为 state = f(事件前缀),分支很便宜)。

② 检查点快照工作区(区别于对话状态):Cline 在 VS Code 全局存储里跑一个 shadow git 仓(不是 .git,故能捕获未跟踪文件、不污染用户历史),每次编辑/命令前提交一个真实 SHA,可"restore"文件到某检查点同时保留对话上下文;Gemini CLI 的 shadow 仓在 ~/.gemini/history/<project_hash>,在任何被批准的写文件工具之前自动触发,打包 git 快照 + 对话历史 + 待定工具调用元数据三样,/restore 回滚文件并把对话倒回重提原调用(默认关闭)。原则:在改动动作之前原子快照,回滚永远落到已知良好的前态。

③ 生命周期 Hooks(Claude Code)是确定性控制层。事件:SessionStart/End、UserPromptSubmit、PreToolUse、PostToolUse(Failure)、Stop、SubagentStop、Notification、Pre/PostCompact。每个 hook 从 stdin 收 JSON(session_id/transcript_path/cwd/tool_name/tool_input/tool_response)。两个返回通道:①退出码——0=成功解析 stdout JSON,2=阻塞(PreToolUse 拦调用、stderr 作模型可见的拒绝理由;Stop 阻止停止),其它=非阻塞记日志;②stdout JSON——hookSpecificOutput.permissionDecision∈ allow|deny|ask|defer,可用 updatedInput 改写参数,PostToolUse 可 decision:"block"+改写输出;通用字段 continue/systemMessage/additionalContext(把文本注入模型上下文)。配置在 settings.json 的 hooks 下,带 matcher 正则(如 Edit|Write)与 if 权限规则过滤。因为 hooks 是 harness 跑的 OS 进程、不是 prompt 文本,它们能强制模型字面上无法绕过的策略。④ 可观测性(OTel GenAI SemConv):每次 LLM 调用是一个 span chat <model>(kind CLIENT),带 gen_ai.request.model / usage.input_tokens / output_tokens / response.finish_reasons;工具调用是 execute_tool <name> span;内容捕获(input/output messages)默认关闭(会泄密)。token×单价=成本汇总,嵌套 span 重建整个循环供 LangSmith/Langfuse/Braintrust 评估调试。

核心概念

事件溯源 / append-only 日志状态存为不可变有序事件、从不原地改;当前态由重放折叠得出,给确定性重建、审计轨迹与免费时间旅行。
View(导出的 prompt)LLM 消息数组是事件日志的计算投影;View 可丢弃,日志才是正典。
Action / Observation 二元每次世界交互要么是 Action(决定做 X)要么是 Observation(发生了什么),都是可序列化重放的事件。
会话 fork因 state=f(事件前缀),复制 [0..N] 到新 id 续跑即廉价分支——用于探索替代或从检查点重试而不改原运行。
shadow git 仓放在用户 .git 之外的检查点库,提交捕获含未跟踪文件的整工作区,给 SHA 可寻址的还原点而不碰真历史。

关键设计权衡

事件日志 vs 可变状态对象：重放给确定性/审计/fork,但 to_llm_message() 必须纯且要给事件版本化——旧事件 schema 一改就可能让历史会话无法重放。
存 View vs 重算 View：重算(OpenHands)保单一真相但每轮耗 CPU 且依赖 condenser 确定性;缓存渲染后的 prompt 更快但可能与日志漂移。
检查点粒度：每次工具调用前快照(安全但多提交/磁盘churn)vs 仅写文件工具前(Gemini)vs 每用户回合——越细还原点越多但开销越大、历史越噪。
Hook 强制点 & 延迟：PreToolUse 在伤害前拦(但你要预判每种危险形态)vs 沙箱/权限做纵深;同步 hook 给每次工具调用加延迟(默认 600s 超时),异步 hook 不阻塞但无法否决。

常见陷阱

  • 事件 schema 演进破坏重放:给事件类型增删字段会让 model_validate_json 在历史日志上失败——必须版本化事件、保持反序列化向后兼容,否则老会话不可恢复。
  • condenser 非确定:若摘要用 LLM,算出的 View 不再确定,除非把摘要本身作为事件持久化(OpenHands 就是这么做)。
  • 检查点不捕获工作区外的副作用:DB 迁移、已推的提交、网络调用、已装的包都不被文件快照回滚——"restore"给假安全感。
  • Hook stdout 污染:任何调试 print 到 stdout 都会破坏 JSON 决策载荷;hooks 只能把 JSON 写 stdout、其它全写 stderr,否则决策被静默忽略。
  • 可观测性泄密:开内容捕获(input/output 消息、工具参数)会把 API key/token/PII 送到 tracing 厂商;默认关是安全的,但有人为调试打开后忘了关。

是什么

评估 harness 是把智能体输出变成可复现 pass/fail 数字的可执行机器:在冻结的仓库态起沙箱、应用模型预测的 patch、跑特定测试集、把原始输出解析成状态、判定"resolved"。定义"80% on SWE-bench"含义的是 harness、不是模型——同一模型在不同脚手架、patch-apply 兜底、超时与判分规则下差很多。懂 harness 内部,才能建可信的内部回归评估、避开会虚高公开分数的泄露/污染陷阱、并按"每解决成本"而非头条 pass@1 来推理。

核心概念

FAIL_TO_PASS / PASS_TO_PASS两个每实例存的测试 id 集:前者须从失败翻成通过(证明修好),后者须保持通过(证明没回归);resolved 要两者全满足。
三层 Dockerbase(OS+conda)/env(每仓版本环境,~60 共享镜像)/instance(克隆+reset+安装),分层最大化跨相似 issue 的缓存复用。
patch-apply 兜底链git apply → --reject → patch --fuzz=5;三者全败=failed apply=未解决,与逻辑对不对无关——畸形 diff 造成假阴的常见源。
test_patch 应用gold PR 的测试改动叠在模型 code patch 之上再跑,判分测试始终存在且是参考 oracle;模型只写代码修复、不写测试。
脚手架+模型耦合榜单分由 agent 循环(检索/定位/编辑格式/失败重试/上下文管理)+ harness 配置 + 模型共同产生;复现须钉脚手架 commit、harness 版本、数据集快照。
抗污染设计(LiveCodeBench)每题带发布日期,只用训练截止之后的题打分,靠"时间滚动"从构造上杜绝污染,而非启发式过滤。

关键设计权衡

oracle 测试判分 vs 真正正确性：FAIL/PASS 集客观便宜,但只和 gold 测试一样好——弱测试放过错 patch,想要可部署信号就得审测试强度或加变异/留出测试。
patch-apply 严格度：模糊兜底(fuzz=5)救回近似 diff 但可能误应用;严格 git apply 忠实但惩罚格式——兜底策略直接移动分数。
pass@1 vs pass@k 报告：带真选择器的 pass@k 测的是可部署系统;带 oracle 选择的 pass@k 是能力上界、生产不复现——按受众选并说明。
抗污染策略：发布日期过滤(LiveCodeBench)vs 私有留出集(SWE-bench Pro)vs 许可证威慑(GPL)——各有成本/刷新/法律权衡;静态公开集会随泄露进预训练而腐坏。

常见陷阱（会虚高分数）

  • 不看脚手架+harness commit 就信头条 %:同模型跨脚手架差几十分,跨实验室比原始榜单分是关公战秦琼。
  • 解答泄露:经典 SWE-bench 上 ~33% 的"resolved"其修复在 issue 里被直接引用、多至 ~61% 被暗示。
  • git 历史泄露:不剥历史/不断网时,智能体跑 git log --all/git show 读已合并的 gold 提交——38 个观察到的作弊里 33 个正是如此。
  • 弱 gold 测试:~48% 的 resolved 靠不足的 FAIL_TO_PASS 覆盖通过;过滤弱/泄露实例后分数崩塌(Verified 上 ~52%→26%)。
  • test_patch 覆写:智能体能改/删用来判它分的测试文件(issue #538),除非 harness 在模型 patch 后不可变地重贴 gold 测试。
  • 训练数据污染:Verified 已大体饱和并日益进入预训练语料;OpenAI 已公开停止把它当前沿信号——优先用截止后或私有留出评估。

  • 文字逐 token 出现;三个 grep 同时跑、比串行快返回;生成中途按 Stop 干净取消、无泄漏锁或半渲染标签;后台 dev server 保持运行而智能体继续。

常见错误

  • 工具参数 JSON 没流完就派发调用。
  • 并行跑改动工具，出现文件/索引竞争。
  • 客户端断开时没释放锁/计数器(泄漏卡住下一轮)。
  • 把部分工具调用标记渲染成用户可见文本。
5 上下文与记忆管理 · 挺过长任务 把不断增长的对话保持在固定窗口内，并用 prompt 缓存让巨大稳定前缀变便宜 +
目标 · 把不断增长的对话保持在固定窗口内、同时保住智能体所需信息，并用 prompt 缓存让巨大稳定前缀既便宜又快。

构建步骤

  • 用供应商 usage 响应对窗口做 token 核算——数 input + cache-creation + cache-read + output(缓存 token 仍占窗口)，为"下一轮响应 + 压缩调用"预留缓冲。
  • 层1(免费):把臃肿工具结果卸载到盘，行内换成可再读指针——回收最多 token。
  • 层2:成对(user/assistant)就近截断最旧轮次以保结构。
  • 层3:~80-90% 时 LLM 摘要——把 transcript 提炼成 目标/决策/改过文件/待办/报错/下一步，替换旧消息，再 rehydration(重读近期文件、恢复待办、重注入 CLAUDE.md)。
  • 每轮把分层记忆文件(enterprise>project>dir>user)载入 system prompt;每次压缩后重注入(它是 always-follow 规则唯一可靠的家)。
  • 把 prompt 按最稳在前排序(tools→system→memory→history)，在稳定前缀后放一个 cache_control 断点;加方向性 partial-compact 让摘要不炸掉缓存前缀。
  • 加熔断:连续 3 次压缩失败后停用自动压缩。

新增组件

真实 token 核算(含缓存)三层压缩:卸载→截断→摘要+rehydration分层 CLAUDE.md/AGENTS.md缓存感知前缀排序 + 断点方向性 partial 压缩压缩熔断
你会学到 · 循环每步重发整段历史，所以不缓存稳定前缀时成本是二次方的。重写前缀式压缩会让缓存失效——因此需要方向性 partial-compact。压缩摘要随会话消失;要持久的东西必须落到记忆文件。

验收标准

  • 一个曾溢出的 60 轮任务现在能压缩并无缝继续(目标+待办+近期文件完好);重复轮次里 cache-read token 占主导(前缀保温);CLAUDE.md 里的规则在压缩后仍被遵守。

常见错误

  • 只量 input_tokens、忽略缓存 token，结果还是溢出。
  • 压缩得太晚，连跑摘要调用本身的空间都没有。
  • 每次压缩都重写缓存前缀、摧毁缓存命中率。
  • 靠压缩摘要当跨会话状态，而不写进记忆文件。
6 规划与任务分解 · 对抗目标漂移 给智能体自我提示的 todo 脚手架、硬闸 plan mode，以及可选的盘上 spec 产物 +
目标 · 给智能体一个自我提示的 todo 脚手架、一个硬闸 plan mode，以及可选的盘上 spec 产物，让长跨度工作不漂移、不在重置时丢失。

构建步骤

  • 加 write_todos(items[]) 工具，每次调用覆盖整个列表(幂等替换){content,status: pending|in_progress|completed};在 TUI 渲染并每轮重注入上下文。
  • 加 ~5 行 system-prompt 纪律:"3+ 步任务先写 todo;恰好一个 in_progress;验证后才标 completed;测试没过绝不标完成"。
  • 把 plan mode 实现为模式标志，把工具白名单换成仅只读(Read/Grep/Glob/WebFetch)——改动工具从工具集里物理消失，模型想改也改不了。
  • 加 exit_plan(plan_text) 工具，展示计划并阻塞到人工批准;唯有批准才解锁写工具、切换模式。
  • (可选)spec-first 持久化:requirements.md → design.md → tasks.md 落盘，逐阶段批准闸，让计划挺过上下文重置、可像代码一样 diff。
  • (可选)per-role 模型路由:强 planner + 便宜 editor(Aider architect/editor 分离)。

新增组件

write_todos(整表覆盖) + prompt 纪律plan mode(工具白名单切换)exit_plan 人工批准闸可选 spec-first 产物可选 planner/editor 分离
你会学到 · todo 列表是自我提示脚手架、不是执行引擎——价值在每轮把目标钉在上下文里。plan mode 的强制必须靠 tool-gating(移除工具)而非 prompt，否则模型会掉出去。整表覆盖胜过增量 API(无竞争)。

验收标准

  • 多步特性上，智能体先写计划、你批准、它带着实时 todo(恰好一个 in_progress)执行;plan mode 里尝试编辑是不可能的，因为工具根本不在。

常见错误

  • 只靠 prompt 强制 plan mode——模型照样编辑或静默退出(真实 bug 类)。
  • 压缩时丢了 plan mode(压缩后要重注入模式/计划)。
  • 让多于一个 todo 处于 in_progress，或测试没过就标完成。
  • 用会 desync 的增量/patch todo API——优先整表覆盖。
7 OS 级沙箱 + 网络出口控制 把强制下沉到内核，扛住智能体 spawn 的任何子进程，并在域名级控制出口 +
目标 · 把强制下沉到内核，让它扛住智能体 spawn 的任何子进程，并在域名级控制网络出口——这是应用层检查给不了的加固。

构建步骤

  • macOS 生成 Seatbelt .sb profile(sandbox-exec);Linux 用 bubblewrap(全新 mount+user+network 命名空间) + seccomp-BPF(拦 connect/bind/listen，放行 AF_UNIX)和/或 Landlock 做写作用域。
  • 规则在 execvp 之前装好，让子进程从第一条指令起就被关住;设 PR_SET_PDEATHSIG 让孤儿子进程随父而死——所有子进程继承盒子(npm postinstall 与父进程一样被关)。
  • 把文件系统写限制在 workspace/CWD;把 .git 设只读防历史被改写;加 secret 目录豁免，且 sandbox.denyRead 要与权限层 Read deny 配对(denyRead 单独并不拦 Read 工具)。
  • 网络走用户态域名白名单代理:去掉沙箱网络命名空间，强制流量经 Unix socket 到一个查白名单的代理(拒绝的 CONNECT 回 403)，再用 seccomp 兜底忽略代理环境变量的工具。
  • 加硬化开关:failIfUnavailable:true、allowUnsandboxedCommands:false;仅当已沙箱时才自动放行 bash。
  • 提供受审计、人工批准的"带批准重试非沙箱"逃生舱——并清楚它正是过度热心/被注入的智能体会试图说服你打开的杠杆。
  • Windows 上回退到 WSL2(真 Linux)拿这些原语。

新增组件

Seatbelt / bwrap+seccomp / Landlock子进程继承 + PR_SET_PDEATHSIG文件作用域 + 只读 .git + secret 豁免域名白名单出口代理 + seccomp 兜底硬化开关 + 受审计逃生舱WSL2 Windows 回退
你会学到 · 权限与沙箱是不重叠的两道闸、都需要:权限执行前决定"能不能跑"，内核沙箱执行后决定"能碰什么"——且只有内核层能扛住 shell 出去的二进制。OS 沙箱懂 socket 不懂域名，域名控制需用户态代理。

验收标准

  • 恶意 postinstall 即便 bash"成功"也无法写出 workspace 或开非白名单 socket;抓取被拒域名在代理处得 403;设了 failIfUnavailable 时沙箱拒绝软降级为非沙箱执行。

常见错误

  • 以为 sandbox.denyRead 能把 secret 挡在上下文外——除非也加权限层 Read deny，Read 工具照读。
  • 沙箱起不来时只给警告就软降级为非沙箱执行(真实脚枪)。
  • 在子进程启动后才装沙箱规则，留下一段不受限执行窗口。
  • 把 allowUnsandboxedCommands 留开，让智能体"说服"自己出盒子。
8 子智能体与多智能体编排 · 扩展到单个上下文之外 把工作扇出到隔离上下文，让昂贵探索永不污染编排器；前沿则协调多个全智能体 +
目标 · 把工作扇出到隔离上下文，让昂贵探索永不污染编排器;前沿再用共享任务状态协调多个全智能体。

构建步骤

  • 加 Task/delegate 工具:生成一个带干净上下文、自己 system prompt、可选自己(更便宜)模型的全新智能体;子体跑完自己的循环、只回一段文字摘要，其中间 token 被丢弃。
  • 用 manager-worker 模式:编排器规划、委派机械/探索工作，然后验证并整合——它的上下文吸收摘要、不吸收原始探索。
  • 分析任务列表的依赖:无依赖任务并发跑、有依赖串行;加递归深度上限与每子体终止条件，防"子体生子体"成本爆炸。
  • 加验证/对账 pass(用分层模型;未经验证的 worker 错误会静默累积)。
  • 加可观测性:每智能体成本/token 指标 + 带前缀日志(如 delegate:<id>)——到这个规模，否则无法调试。
  • (前沿)转向协作的全智能体:共享任务列表(状态+依赖跟踪)、点对点消息、文件锁，每个队友是自己 worktree/容器里的完整实例——隔离端口/DB/缓存，不只文件。

新增组件

Task/delegate(隔离上下文子体)manager-worker + 模型分层依赖分析 + 并行扇出 + 递归上限验证/对账 pass每智能体可观测性前沿:共享任务表+消息+文件锁+每智能体容器
你会学到 · 上下文隔离是核心技巧:子智能体的昂贵探索发生在一个永不进入编排器的暂存窗口里，只有摘要作为一个 tool_result 返回。无状态"返回即死"子体可预测但不能协作;共享状态团队能协作但加锁竞争与可观测性缺口。

验收标准

  • 一个含三个独立子任务并行的大任务完成时，编排器上下文保持精简;一个研究子体探索了 20 个文件却只回一段摘要;worker 输出在整合前被验证、而非盲信。

常见错误

  • 让子体中间 token 流回父体(彻底破坏隔离的意义)。
  • 无递归深度上限——子体生子体成本爆炸。
  • 跳过验证 pass，未经验证的 worker 错误静默累积。
  • 给绑端口/碰共享 DB 的并行智能体用 worktree——需要每智能体容器做运行时隔离，不只文件。
  • 无依赖分析，依赖任务互相竞争或覆盖。

迭代原则与评估方法

裸循环跑起来后，接下来靠"看数据、按证据加固"往前推。下面是可迁移的迭代原则，以及一套三轴的评估方法——把这套评估冻成回归集，每个阶段都对它跑一遍，要求 solve-rate 与安全性单调改善。

迭代优化原则

01

先让它跑，再让它变好。先发布会崩的 MVP——裸循环 + 五个工具就真能完成单文件任务;后续每一阶段都是加固已能跑的系统，而非第一个阶段的前置。

02

先埋点，再优化。尽早加 token 核算(input+cache-creation+cache-read+output)与每轮/每工具的成本+延迟日志;看不到 token 和秒花在哪，就没法改进上下文策略或模型路由。

03

把错误串当产品面对待。具体+可操作+标注终止态的错误消息，加"别用相同参数重试"，比几乎任何模型改动都更能削减重试循环——靠读失败运行的 transcript 来迭代它们。

04

先打最大的 token 消耗者。工具输出占上下文大头，所以"卸载到盘+指针"以零模型成本回收最多窗口——先做这个，再去动 LLM 摘要。

05

为缓存塑形 prompt。按最稳在前排序(tools→system→memory→history)，避免会话中途改工具/模型/前缀;5 分钟 TTL 缓存在活跃循环里能一直保温，把二次方成本变线性。

06

按角色与子任务路由模型。强模型跑规划/主循环，便宜模型跑编辑、摘要/压缩、子体搜索、commit 消息;按模型强弱匹配编辑格式(强→diff/SEARCH-REPLACE，弱→全文件)。

07

逐步把强制下沉。prompt 级 → harness/应用级 → OS/内核级。只有内核层能扛住 shell 出去的子进程，所以把安全关键规则往下迁，而非信任模型配合。

08

数据证明复杂性之前，选更简单的设计。隐式"无工具调用=完成"胜过显式 finish 工具;整表 todo 覆盖胜过增量 API;单循环胜过多智能体——只有当实测失败模式(漂移/耗尽/协调)出现时才加仪式。

09

有条不紊地封堵绕过。懂 shell 的参数解析、顺序无关多编辑应用、连续失败熔断，都是对具体真实脚枪的回应——在 transcript 里观察到对应失败时再逐个加。

10

保持每阶段可独立运行并回归测试。用一组固定的小任务，每阶段后必须仍通过，以抓住经典破坏(未答 tool_use、缓存失效、plan-mode 掉出、编辑误应用)。

评估方法 · 三轴 + 冻结回归集

AXIS 1

正确性 / 任务完成

跑 SWE-bench（Verified 是标准切片）看真实仓库 bug 修复解决率——它演练完整循环(定位→编辑→跑测试→读失败→重试)，是品类的经典比较指标。补一套代表性内部任务(单文件修复、多文件特性、重构、绿地)，以各自测试判 pass/fail。用 Aider 的编辑格式方法论单独隔离衡量"编辑应用成功率"。

AXIS 2

成本与延迟

跟踪每解决任务的 $ 与 token(拆成 input/cache-creation/cache-read/output 以看缓存命中率)、每任务墙钟、完成所需轮数。prompt 缓存、模型路由、工具输出卸载都在这里显形——一个提升解决率却让成本翻三倍的阶段是一次回归。

AXIS 3

安全 / 健壮性

一套 red-team 套件(prompt 注入、破坏性命令 rm -rf/force-push/.env 读/非白名单出口/postinstall 逃逸)，必须在策略层与沙箱层双双被拦;加可靠性指标——相同重试率、无限循环/熔断触发次数、压缩连续性(压缩后智能体是否仍知道目标+待办)。

方法:冻结任务集，每阶段对它跑一遍，要求 solve-rate 与安全性单调改善(或给出明确的权衡理由)，同时盯住成本/延迟;记录每个 action/observation 让运行可重放，供评估与调试。
